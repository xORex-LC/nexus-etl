# Архитектурные инварианты — nexus-etl

Формальные инварианты, которые **нельзя нарушать**. Каждый инвариант поддерживается хотя бы одним автоматическим механизмом. Задача считается незавершённой, пока нарушенный инвариант не восстановлен.

---

## Как читать этот документ

| Символ | Значение |
|---|---|
| ✅ | Проверяется автоматически (CI/lint) |
| ⚠️ | Поддерживается конвенцией + code review |
| ❌ | Не проверяется — только дисциплина |
| — | Неприменимо |

### ИНВАРИАНТ I-01: Domain не импортирует infra и delivery

**Утверждение:** Любой модуль в `connector/domain/` или `connector/usecases/` не должен импортировать ничего из `connector/infra/` или `connector/delivery/`.

**Почему:** Domain-слой определяет бизнес-логику и порты (Protocols). При импорте конкретных infra-реализаций domain связывается с деталями I/O — тогда невозможно подменять адаптеры, тестировать в изоляции и рассуждать о бизнес-правилах без инфраструктурного контекста.

**Сценарий нарушения:** Domain-сервис напрямую импортирует `SqliteEngine` для выполнения запроса. Теперь тесты бизнес-логики требуют реального SQLite-файла, а смена на PostgreSQL влечёт правки в domain-файлах.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `lint-imports` (import-linter) | ✅ | Контракт определён в `pyproject.toml` |
| `pytest -m architecture` | ✅ | Архитектурный тест-сьют |
| PostToolUse hook | ✅ | Запускается после каждого Edit/Write |
| Code review | ⚠️ | Вторичный рубеж |

**Известные исключения:** DI-контейнеры в `delivery/cli/containers.py` объединяют все слои — это единственная намеренная точка, где встречаются все уровни архитектуры.

### ИНВАРИАНТ I-02: SQLite доступен только через SqliteEngine

**Утверждение:** Весь доступ к SQLite должен идти через `connector/infra/sqlite/engine.py:SqliteEngine`. Прямое использование stdlib-модуля `sqlite3` за пределами самого `SqliteEngine` запрещено.

**Почему:** `SqliteEngine` обеспечивает WAL-режим, busy timeout, foreign keys и управление жизненным циклом транзакций (DEFERRED/IMMEDIATE/EXCLUSIVE). Обход ведёт к несогласованному состоянию БД, отсутствию прагм и бесследной потере владельца транзакции.

**Сценарий нарушения:** Разовый `sqlite3.connect()` в хендлере пропускает WAL-режим. При конкурентном доступе база блокируется без таймаута, а запись молча обрезает данные.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `grep -r "import sqlite3"` | ⚠️ | Ручной аудит; автоматической блокировки нет |
| Code review | ⚠️ | Должен ловить прямое использование `sqlite3` |
| Кастомное правило `ruff` | ❌ | Не настроено |

**Известные пробелы:** Жёсткой автоматической проверки нет. Рекомендуется добавить banned-import правило в `ruff` или grep в CI.

### ИНВАРИАНТ I-03: Секреты не записываются вне vault

**Утверждение:** Значения секретов не должны появляться в: лог-файлах, report JSON (`reports/`), plan-файлах (`var/plan_*.json`), дампах переменных окружения или любых файлах вне `connector/infra/secrets/`.

**Почему:** Plan-файлы разделяются между `nexus import plan` и `nexus import apply` и могут храниться на диске часами или днями. Логи часто отправляются на observability-платформы. Любая утечка секретов в этих артефактах компрометирует безопасность учётных данных.

**Сценарий нарушения:** Разработчик добавляет `logger.debug("payload: %s", payload)` до вызова `PayloadSanitizer`. Пароли попадают в `var/logs/import_apply_*.log`.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `PayloadSanitizer` | ✅ | Применяется перед каждой записью в отчёт |
| `TargetSafeLogger` | ✅ | Применяется перед каждым HTTP-логом |
| Схема `PlanItem.secret_fields` | ✅ | Только имена полей, не значения |
| Сканирование секретов в CI | ❌ | Не настроено (`gitleaks`, `truffleHog`) |
| Code review | ⚠️ | Обязателен аудит новых log-вызовов рядом с payload/secret-объектами |

**Известные пробелы:** Автоматическое сканирование секретов в CI не настроено.

### ИНВАРИАНТ I-04: Транзакции SqliteEngine не вкладываются

**Утверждение:** Вызов `engine.transaction()` внутри уже активной транзакции того же экземпляра engine бросает `RuntimeError`. Никакой код не вправе подавлять или перехватывать эту ошибку, чтобы повторить попытку или проигнорировать её.

**Почему:** SQLite не поддерживает истинные вложенные транзакции через `BEGIN`. Вложение в некоторых драйверах молча повышает внутренний `BEGIN` до savepoint, однако текущий `SqliteEngine` savepoint'ы не реализует. Разрешение вложенности маскирует баги в области откатов.

**Сценарий нарушения:** Роль кэша открывает `transaction()` и затем вызывает хелпер, который тоже открывает `transaction()`. `RuntimeError` перехватывается и подавляется. Откат внутренней транзакции теряется, оставляя наполовину записанные строки.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Флаг `SqliteEngine._active` | ✅ | Бросает `RuntimeError` при вложенном входе |
| Unit-тесты для `SqliteEngine` | ✅ | Покрывают случай вложенной транзакции |
| Code review | ⚠️ | Запрещены `except RuntimeError: pass` рядом с транзакциями |

### ИНВАРИАНТ I-05: polars ограничен слоем infra

**Утверждение:** `import polars` допустим только в файлах под `connector/infra/` (sources, dictionaries, backends). Domain и usecases работают исключительно с plain Python `dict`.

**Почему:** `polars` — тяжёлая зависимость, привязанная к C-расширениям и DataFrame. Его проникновение в domain связывает бизнес-правила с конкретным представлением данных: unit-тесты начинают требовать конструирования DataFrame, а изменения polars API ломают логику нормализации.

**Сценарий нарушения:** Правило стадии normalize импортирует `polars` для regex на Series. Теперь unit-тесты нормализации требуют установленного polars, и любое изменение API polars может сломать логику нормализации.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Контракт `lint-imports` | ✅ | Если настроен в `pyproject.toml` |
| Code review | ⚠️ | Вторичный рубеж |
| `grep -r "import polars" connector/domain connector/usecases` | ⚠️ | Ручной аудит |

### ИНВАРИАНТ I-06: Секреты записываются в enrich, читаются в apply

**Утверждение:** Жизненный цикл секрета строго такой: **запись в vault на стадии enrich → ссылка (по ключу) в PlanItem → чтение из vault на стадии apply**. Ни одна стадия не вправе записывать значение секрета в поле PlanItem, и ни одна domain-стадия не вправе читать расшифрованное значение секрета.

**Почему:** Этот порядок гарантирует, что секреты хранятся зашифрованными между планированием и выполнением. Обращение порядка (чтение в enrich с кэшированием значения) раскрывало бы расшифрованные секреты в памяти на протяжении всего planning-конвейера и потенциально сериализовало бы их в plan JSON.

**Сценарий нарушения:** `EnricherCore` сохраняет расшифрованный пароль в обогащённый dict. `PlanBuilder` сериализует его в `plan.json`. Файл случайно попадает в git.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Схема `PlanItem` | ✅ | `secret_fields: list[str]` — только имена полей |
| Расположение `VaultReadService` | ✅ | Подключён только в `ImportApplyService` (фаза apply) |
| Расположение `SecretLocatorService` | ✅ | Подключён только в `EnricherCore` (фаза enrich, запись) |
| Code review | ⚠️ | Обязателен аудит новых enrich/apply-подключений в DI-контейнерах |

### ИНВАРИАНТ I-07: Все коды ошибок зарегистрированы в ErrorCatalog

**Утверждение:** Каждая строка диагностического кода, используемая в `DiagnosticItem`, должна быть зарегистрирована в `connector/domain/diagnostics/core_catalog.py` через `build_core_catalog()`. Произвольные строковые литералы в качестве кодов ошибок запрещены.

**Почему:** Централизованный каталог — единственный источник истины для severity, системной классификации и stop policy. Незарегистрированные коды обходят `StopPolicy`, дают неопределённую severity в отчётах и делают отладку непредсказуемой.

**Сценарий нарушения:** Стадия создаёт `DiagnosticItem(code="MY_CUSTOM_ERR", ...)` без регистрации. `StopPolicy` возвращает `False` (нет записи → нет остановки), и фатальная ошибка молча продолжает пайплайн.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `DiagnosticItem.from_catalog()` | ✅ | Бросает `KeyError`, если код отсутствует в каталоге |
| Unit-тесты каталога | ⚠️ | Должны проверять резолвинг всех кодов через `from_catalog()` |
| Code review | ⚠️ | Новые коды обязательно появляются в `core_catalog.py` |

### ИНВАРИАНТ I-08: httpx ограничен infra/target/transports/http/

**Утверждение:** `import httpx` допустим только в файлах под `connector/infra/target/transports/http/`. Никакой другой слой (domain, usecases, delivery, другие infra-модули) не вправе импортировать httpx.

**Почему:** Изоляция HTTP-клиента не даёт HTTP-деталям (таймауты, SSL, connection pooling, redirect-политики) проникать в бизнес-логику или другие infra-компоненты. Это позволяет менять транспорт, не трогая никакой другой код.

**Сценарий нарушения:** Use case импортирует `httpx` для быстрой health-проверки. Теперь usecases зависят от сети — тесты становятся медленными и невозможными без интернета.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Контракт `lint-imports` | ✅ | Если настроен для httpx |
| Code review | ⚠️ | Вторичный рубеж |
| `grep -r "import httpx" connector/` | ⚠️ | Ручной аудит |

### ИНВАРИАНТ I-09: TransformResult.row равен None тогда и только тогда, когда errors непустой

**Утверждение:** `TransformResult[T].row is None` ⟺ `len(result.errors) > 0`. Результат с `row=None` и пустым errors недопустим. Результат с непустым `row` и непустым errors тоже недопустим.

**Почему:** Потребители downstream используют `if result.row is None` как канонический сигнал для пропуска обработки. При нарушении инварианта либо валидные строки отбрасываются (None без ошибок), либо невалидные обрабатываются (non-None с ошибками).

**Сценарий нарушения:** Стадия возвращает `TransformResult(row=some_obj, errors=(err,))`. Шаг apply обрабатывает строку, отправляет в target, а `err` молча исчезает из отчёта.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `TransformResult.__post_init__` | ✅ | Валидирует инвариант при конструировании |
| Unit-тесты для `TransformResult` | ✅ | Покрывают граничные случаи |
| Аннотации типов | ⚠️ | `row: T | None` типизирован корректно |

### ИНВАРИАНТ I-10: tenacity ограничен infra/target/core/engines/retry_engine.py

**Утверждение:** `import tenacity` допустим только в `connector/infra/target/core/engines/retry_engine.py`. Retry-логика в других местах должна вызывать `TargetRetryEngine`, а не tenacity напрямую.

**Почему:** Параметры retry (backoff, jitter, max attempts) настраиваются через `RetryConfig` в `TargetSpec`. Распыление tenacity по кодовой базе делает retry-поведение невозможным для аудита, настройки или единообразного отключения.

**Сценарий нарушения:** Хелпер cache refresh использует `@tenacity.retry(stop=stop_after_attempt(3))` напрямую. При тестировании retry утраивает время выполнения теста без способа отключить это через конфиг.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Code review | ⚠️ | Жёсткой автоматической проверки нет |
| `grep -r "import tenacity"` | ⚠️ | Ручной аудит |

### ИНВАРИАНТ I-11: PlanItem.secret_fields содержит только имена полей

**Утверждение:** `PlanItem.secret_fields` — это `list[str]`, список имён полей, чьи значения хранятся в vault. Список не должен содержать сами значения секретов, vault-ключи или зашифрованные блобы.

**Почему:** Plan-файлы (`var/plan_*.json`) могут храниться на диске между планированием и выполнением. Если бы они содержали реальные секреты, любое чтение файла раскрывало бы учётные данные. Дизайн хранит только имена полей, требуя vault-lookup в момент apply для получения значения.

**Сценарий нарушения:** Sink compiler сериализует `{"secret_fields": ["password"], "password": "s3cr3t"}` в PlanItem. Файл plan читаем любым OS-пользователем с доступом к `var/`.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Схема `PlanItem` | ✅ | Поле типизировано как `list[str]` — хранение значений невозможно |
| `PayloadSanitizer` | ✅ | Удаляет значения secret-полей перед записью плана |
| Code review | ⚠️ | Аудит вывода sink compiler при добавлении новых secret-полей |

### ИНВАРИАНТ I-12: DSL op chain останавливается на первой ошибке

**Утверждение:** Когда `TransformationEngine.apply(value, ops)` встречает исключение в любой операции, цепочка немедленно останавливается, возвращается `DslIssue(code="DSL_OP_FAILED")`, последующие операции не выполняются.

**Почему:** Операции в цепочке семантически зависят друг от друга (например, `trim` → `to_int` → `format_mask`). Продолжение после сбоя даёт неопределённые промежуточные значения, которые могут молча коррумпировать последующие операции или производить неверный вывод без сигнала ошибки.

**Сценарий нарушения:** `to_int` бросает `ValueError` на `"N/A"`. Цепочка продолжается в `format_mask` с исходной строкой. Результат — `"N/A"`, отформатированный как номер телефона: ошибки нет, данные неверные.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| Реализация `TransformationEngine.apply()` | ✅ | Перехватывает все исключения, останавливается на первом |
| Unit-тесты для `TransformationEngine` | ✅ | Покрывают поведение halt-on-error |
| Code review | ⚠️ | Запрещены `continue` или swallow-логика внутри engine |

### ИНВАРИАНТ I-13: DI-контейнеры — единственная точка создания infra-объектов

**Утверждение:** Инфраструктурные объекты (репозитории, gateway'и, engine'ы, провайдеры) должны создаваться только внутри `connector/delivery/cli/containers.py` (или sub-контейнеров, которым он делегирует). Domain и usecases никогда не вызывают `SqliteEngine(...)`, `SqliteVaultRepository(...)` или аналогичные конструкторы напрямую.

**Почему:** DI-контейнеры управляют жизненным циклом (singleton vs. factory), инжектируют правильный конфиг и позволяют подменять реализации в тестах. Прямое конструирование обходит инжекцию конфига, создаёт неконтролируемые экземпляры и делает граф зависимостей непрозрачным.

**Сценарий нарушения:** Use case напрямую создаёт `CsvSourceReader(path)` с хардкоженным путём. Интеграционные тесты не могут подставить `tmp_path`-based reader и обращаются к реальной файловой системе.

**Контроль:**

| Механизм | Статус | Примечание |
|---|---|---|
| `lint-imports` (domain/usecases не могут импортировать infra) | ✅ | Блокирует прямое конструирование infra в domain |
| Code review | ⚠️ | Проверять `delivery/commands` на случайные infra-конструкторы |
| Аннотации типов конструкторов use case | ⚠️ | Должны принимать только Protocol-типы |
