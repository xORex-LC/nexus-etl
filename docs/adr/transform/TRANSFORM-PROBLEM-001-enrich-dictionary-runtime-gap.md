# TRANSFORM-PROBLEM-001: Отсутствует runtime-реализация справочников для enrich lookup

> **Статус**: Открыта (кандидат решения: TRANSFORM-DEC-001)
> **Дата создания**: 2026-02-19
> **Затронутые компоненты**: `DictionaryProviderPort`, `ProviderGateway`, `TransformProviderDeps`, `EnricherEngine`

---

## 📋 Контекст

В transform-домене уже есть порт `DictionaryProviderPort`, а в enrich-провайдерах
зарегистрирован `dictionary.by_key`. Это позволяет декларативно описывать lookup в DSL.

Однако у порта нет production-ready адаптера. В результате архитектурный контракт есть, но
исполняемой справочной подсистемы для enrich нет.

Параллельно есть нефункциональное требование: словари могут быть очень большими (миллионы
строк, десятки/сотни колонок), при этом enrich обычно использует 1-2 поля на правило.
Нужен движок, оптимизированный под column projection и быстрый lookup без подъема внешней БД.

---

## ⚠️ Проблема

Enrich-слой не имеет готового runtime-источника для `dictionary.by_key`, поэтому справочные
lookup-правила декларативно описываются, но не обеспечены рабочей инфраструктурой.

В текущем виде:
- порт существует только как protocol-контракт;
- lifecycle словарей (загрузка, версия, обновление) не определен;
- производительная выборка ограниченного набора колонок не зафиксирована архитектурно.

---

## 🔍 Симптомы

- **Симптом 1**: При использовании `dictionary.by_key` выполнение enrich зависит от
  отсутствующего адаптера и ломается на wiring/runtime этапе.
- **Симптом 2**: Команды вынуждены обходить enrich-lookup через cache/ручные костыли, что
  размывает границы ответственности transform-слоя.
- **Симптом 3**: Нет единого формата регистрации словарей (где файл, какой ключ, какие
  колонки/нормализаторы, temporal-поле).
- **Симптом 4**: Нет стандарта для производительной работы с wide-таблицами при выборке
  ограниченного набора полей.

---

## 📊 Масштаб проблемы

- **Частота**: Всегда при попытке использовать справочники в enrich.
- **Критичность**: Высокая (блокирует полноценный словарный enrich).
- **Затронуто**: Transform pipeline (enrich), Dataset DSL для lookup-правил, infra wiring.

---

## 🧪 Как воспроизвести

1. Описать enrich lookup-правило с `provider.name: dictionary.by_key`.
2. Запустить transform pipeline без реального `DictionaryProviderPort` адаптера в deps.
3. Дойти до enrich стадии и выполнить lookup-правило.
4. **Ожидаемый результат**: Провайдер находит запись в словаре и возвращает данные для
   обогащения.
5. **Фактический результат**: Lookup path не имеет реальной backend-реализации, enrich не
   может выполнить справочный запрос.

---

## 🚫 Почему это проблема?

- Порт без адаптера не дает бизнес-ценности и создает ложное ощущение готовности фичи.
- Lookup-обогащение является ключевым сценарием transform; без него часть DSL неработоспособна.
- Отсутствует единая точка для требований производительности на больших словарях.
- Растет архитектурный долг: команды компенсируют пробел локальными обходными решениями.

---

## 💡 Возможные решения (обсуждение)

### Вариант 1: SQLite-справочники с индексами
- **Идея**: Реализовать `DictionaryProviderPort` поверх SQLite.
- **Плюсы**: Простая эксплуатация, знакомый стек.
- **Минусы**: row-store модель хуже подходит под колонночные выборки на больших справочниках;
  требуется аккуратное индексирование и ручной тюнинг под lookup-нагрузку.

### Вариант 2: DuckDB как основной dictionary backend
- **Идея**: Реализовать адаптер на DuckDB с projection/predicate pushdown.
- **Плюсы**: Хорошая производительность на wide-таблицах, SQL API, file-based формат без
  отдельного сервера.
- **Минусы**: Появляется новый runtime dependency и операционные практики его использования.

### Вариант 3: Polars backend
- **Идея**: Использовать Polars (Lazy/DataFrame) для lookup и batch join сценариев.
- **Плюсы**: Сильная векторизация и хорошая работа с Arrow/Parquet.
- **Минусы**: Для поштучных lookup нужен аккуратный runtime design (иначе высокий overhead
  при частых мелких запросах).

### Вариант 4: Backend-pluggable подсистема (DuckDB + Polars)
- **Идея**: Единый адаптер порта и сменные backend-стратегии под разные workload.
- **Плюсы**: Гибкость и эволюция без изменений доменного контракта.
- **Минусы**: Больше архитектурной сложности на старте.
