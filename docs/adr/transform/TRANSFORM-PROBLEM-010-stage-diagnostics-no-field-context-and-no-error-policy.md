# TRANSFORM-PROBLEM-010: Stage diagnostics не фиксируют поле и значение при ошибке — нет декларативной политики обработки ошибок полей

> **Статус**: Решена в TRANSFORM-DEC-009
> **Дата создания**: 2026-02-27
> **Затронутые компоненты**: `TransformationEngine`, `DslIssue`, `DiagnosticItem`, `NormalizerCore`, `MapperCore`, `EnricherCore`, `MatchCore`, `ResolveCore`, `append_dsl_issue`, `employees.normalize.yaml`, `employees.mapping.yaml`, `employees.enrich.yaml`

---

## 📋 Контекст

В пайплайне трансформации пять стадий (MAP → NORMALIZE → ENRICH → MATCH → RESOLVE), каждая из которых обрабатывает поля записи через DSL-операции. При возникновении ошибки на любой стадии создаётся `DiagnosticItem`, который записывается в отчёт и логи. Эти события должны быть полезны операторам для отладки и трассировки проблем в данных.

Параллельно ведётся работа по устранению `diagnostic_catalog.py` — per-dataset каталога ошибок с per-field кодами. Анализ показал, что этот механизм — неправильный способ решения задачи: он завязывает поведение на конкретный датасет и порождает мёртвые коды.

---

## ⚠️ Проблема

**1. Диагностика не содержит контекст поля.**

`TransformationEngine.apply()` при ошибке операции создаёт `DslIssue(field=None)` — движок не знает, для какого поля запускались операции. Вызывающий код (например, `normalizer_core`) знает поле (`rule.field`), но не передаёт его в диагностику. В итоге в отчёте и логах: «DSL_OP_FAILED», но без указания поля — невозможно понять, что именно пошло не так.

**2. Диагностика не содержит проблемное значение.**

Даже если поле известно, сам значение, которое вызвало ошибку (например, строка `"abc"` при попытке `int_if_digits`), нигде не фиксируется. Оператор видит код ошибки без данных для воспроизведения.

**3. Политика обработки ошибок — хардкод, не декларация.**

Поведение при ошибке (пропустить запись / продолжить / обнулить поле) задаётся строкой `on_error: "warn"` в YAML, которая обрабатывается через `if on_error == "warn"` в `append_dsl_issue`. Это:
- поддерживает только один вариант отклонения от дефолтного поведения (`warn`)
- не позволяет задать разное поведение для разных полей одной стадии
- хардкодит строку без enum/validation

**4. Нет единого контракта между стадиями.**

Каждая стадия решает вопрос по-своему:
- `normalize.yaml` — `on_error: warn` (string, stage-level)
- `enrich.yaml` — `on_error: warn` (string, per-generator)
- `resolve.yaml` — `on_unresolved: pending` (другое имя, другая семантика)
- MAP — нет `on_error` вообще

---

## 🔍 Симптомы

- **Симптом 1**: В отчёте/логах при ошибке NORMALIZE или MAP: `field=None` — невозможно определить, какое поле данных вызвало проблему.
- **Симптом 2**: Нет возможности указать в YAML для поля `phone`: «если trim упал — продолжить с None», а для поля `organization_id`: «если `int_if_digits` упал — блокировать запись».
- **Симптом 3**: `diagnostic_catalog.py` содержит per-dataset коды (`INVALID_EMAIL`, `INVALID_BOOLEAN`) как компенсацию отсутствия field-context в generic диагностике — мёртвые коды, которые никогда не эмитировались.
- **Симптом 4**: `on_error: "warn"` проверяется строкой в `append_dsl_issue` — любая опечатка в YAML создаёт молчащую регрессию.

---

## 📊 Масштаб проблемы

- **Частота**: Всегда — при любой ошибке в операциях DSL field-контекст теряется.
- **Критичность**: Средняя — пайплайн работает, но диагностируемость низкая; при росте количества полей и датасетов отладка становится непрактичной.
- **Затронуто**: Все 5 стадий трансформации, все датасеты.

---

## 🚫 Почему это проблема?

- Отчёты о трансформации не actionable: оператор знает, что запись отклонена, но не знает почему и в каком поле.
- Политика поведения при ошибке не может конфигурироваться per-field в YAML: нельзя задать «для этого датасета поле X некритично, а поле Y — блокирует».
- `diagnostic_catalog.py` с per-dataset кодами — попытка компенсировать отсутствие field-context; этот механизм нужно удалить, а field-context получать декларативно из DSL.
- Разнородный `on_error` (строка / разные имена в разных стадиях) усложняет поддержку и понимание поведения пайплайна.

## 💡 Возможные решения (обсуждение)

### Вариант 1: Передать field через `engine.apply(field=...)`

- **Идея**: Добавить параметр `field` в `TransformationEngine.apply()`, движок сам вставляет его в `DslIssue`.
- **Плюсы**: Минимальные изменения.
- **Минусы**: Движок начинает знать о семантике поля; не решает проблему политики; `value` не передаётся.

### Вариант 2: Декларативная политика ошибок в YAML + обогащение на уровне ядра стадии

- **Идея**: Движок остаётся универсальным. Ядро стадии (normalizer_core и др.) знает и поле, и значение, и политику из YAML. После `engine.apply()` ядро само обогащает `DslIssue` (field, value, action) и применяет действие по политике.
- **Плюсы**: Движок остаётся чистым; политика полностью декларативна в YAML; field/value попадают в диагностику; каждое поле может иметь собственное поведение.
- **Минусы**: Больше изменений — затрагивает несколько ядер стадий и YAML-схемы.

**Принято**: Вариант 2. Решение зафиксировано в [TRANSFORM-DEC-009](./TRANSFORM-DEC-009-field-error-policy-and-diagnostic-enrichment.md).
