# OBSERVABILITY-PROBLEM-001: Непоследовательное использование logging и structlog

> **Статус**: Решена в OBSERVABILITY-DEC-001
> **Дата создания**: 2026-02-19
> **Затронутые компоненты**: `connector/infra/logging/setup.py`, все модули использующие `logging.getLogger()`

---

## 📋 Контекст

В проекте есть два сосуществующих подхода к логированию:
1. `connector/infra/logging/setup.py` — stdlib-based, ручная передача `runId`/`component` через `extra={}`
2. `structlog` — уже добавлен как зависимость и используется в ряде мест

Это не баг — оба подхода работают. Но их сосуществование создаёт инфраструктурные издержки.

---

## ⚠️ Проблема

Два механизма логирования дают несогласованный опыт разработки и эксплуатации:

- **Контекст вручную**: `logEvent(logger, level, runId, component, message)` и `EnsureFieldsFilter`
  решают задачу, которую structlog закрывает одним `bind_contextvars(run_id=..., component=...)` в
  точке старта
- **Разные форматы**: часть логов идёт через stdlib-форматтер `%(runId)s comp=%(component)s msg=%(message)s`,
  часть — через structlog-процессоры; в prod-логах они выглядят по-разному
- **Дрейф**: каждый новый модуль выбирает один из двух подходов на своё усмотрение

---

## 🔍 Симптомы

- В одном pipeline часть строк имеет `run_id` в контексте, часть — нет
- `EnsureFieldsFilter` существует только для того, чтобы форматтер не падал с `KeyError` —
  это workaround, а не архитектурное решение
- `createCommandLogger` дублирует конфигурацию, которую structlog делает глобально один раз

---

## 📊 Масштаб проблемы

- **Частота**: При каждом новом модуле с логированием
- **Критичность**: Низкая (логирование работает), но растёт с каждым новым компонентом
- **Затронуто**: Все команды, все новые use cases
