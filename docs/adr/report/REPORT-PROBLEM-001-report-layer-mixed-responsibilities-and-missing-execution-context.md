# REPORT-PROBLEM-001: Report layer смешивает ответственности и не имеет единого Execution Context

> **Статус**: Закрыто
> **Дата создания**: 2026-03-01
> **Затронутые компоненты**: `run_with_report`, `ReportCollector`, `ApplyReportPresenter`, `report_writer`

---

## 📋 Контекст

Report слой эволюционировал итеративно: сначала как JSON-артефакт в конце команды, затем как набор точечных интеграций из runtime/usecase/presenter.  
В результате добавление новой телеметрии (cache lookup, dictionary runtime, vault rollout, lifecycle errors) выполняется в разных местах и разными способами.

Цель на текущем этапе: перейти к архитектуре, где отчёт — это агрегированная проекция событий выполнения, а не набор ad-hoc мутаций в нескольких слоях.

---

## ⚠️ Проблема

В report layer нет единого scoped-контекста выполнения и единой ingestion-модели событий.  
Сейчас одновременно присутствуют:

- runtime-level mapping ошибок/результатов;
- row-level запись из usecases;
- прямые мутации summary/items/status в presenter;
- финализация и рендер в отдельном инфраструктурном модуле.

Это создаёт архитектурное смешение orchestration, aggregation и rendering.

---

## 🔍 Симптомы

- **Симптом 1**: Разные подсистемы пишут в отчёт через разные контракты (`add_item`, `set_context`, прямые мутации полей коллектора).
- **Симптом 2**: Для новых источников телеметрии нет единой точки интеграции; приходится вручную решать, где и как писать данные.
- **Симптом 3**: Логика derive/report status и runtime exit semantics исторически расходилась и требует точечных фиксов.
- **Симптом 4**: Структура контекста основана на строковых ключах без типизированной event/schema-модели.

---

## 📊 Масштаб проблемы

- **Частота**: Всегда при эволюции report-слоя и подключении новых подсистем.
- **Критичность**: Высокая.
- **Затронуто**: CLI runtime команды (`mapping`, `normalize`, `enrich`, `match`, `resolve`, `import plan/apply`, `cache*`, `check-api`), observability и post-run анализ отчётов.

---

## 🧪 Как воспроизвести

1. Добавить в любую runtime-команду новый блок технической телеметрии подсистемы (например, lookup/cache metrics).
2. Попробовать сделать это единообразно для `enrich`, `resolve` и `cache-refresh`.
3. Обнаружить, что точки записи и контракт данных различаются (handler/usecase/presenter/runtime).
4. **Ожидаемый результат**: единый ingestion API и scoped execution context, куда пишутся события.
5. **Фактический результат**: телеметрия распределяется по нескольким слоям и форматам, что повышает coupling и стоимость изменений.

---

## 🚫 Почему это проблема?

- Нарушаются границы ответственности: delivery orchestration, доменная агрегация и форматный рендер пересекаются.
- Трудно масштабировать детализацию отчёта без роста условной логики в runtime/usecase/presenter.
- Сложно гарантировать консистентность и воспроизводимость структуры отчёта между командами.
- Появляется риск регрессий при расширении схемы report/context.

---

## 💡 Возможные решения (обсуждение)

> Этот раздел может содержать первоначальные идеи до принятия финального решения

### Вариант 1: Продолжить эволюцию текущего ReportCollector
- **Идея**: Дальше расширять текущий collector и точечные runtime/presenter адаптеры.
- **Плюсы**: Минимальный upfront cost.
- **Минусы**: Не устраняет root-cause смешения ответственности, сложность будет накапливаться.

### Вариант 2: Execution Context + Event ingestion + Assembler/Renderer (целевой)
- **Идея**: Ввести scoped `IReportContext`, единый `IReportSink.emit(event)`, сборку отчёта через `ReportAssembler` и рендер через `IReportRenderer`.
- **Плюсы**: Чёткие границы, расширяемость, типизированные события, единый lifecycle.
- **Минусы**: Требует этапной миграции и адаптеров совместимости.

### Вариант 3: Полный автоматический tracing без явной записи
- **Идея**: Собирать всю телеметрию только прозрачными декораторами/интерцепторами.
- **Плюсы**: Меньше кода в сценариях.
- **Минусы**: Риск “магии”; бизнес-события сложнее контролировать и объяснять.
