# VAULT-PROBLEM-001: Plaintext dev-vault и отсутствующий production-контур секретов

> **Статус**: Решена в [VAULT-DEC-001](./VAULT-DEC-001-envelope-encrypted-vault-with-hexagonal-ports.md)
> **Дата создания**: 2026-02-18
> **Затронутые компоненты**: `SecretProviderProtocol`, `SecretStoreProtocol`, `FileVaultSecretStore`, `FileVaultSecretProvider`, `OperationApplyAdapter`, `EnricherCore`, `build_secret_provider()`

---

## 📋 Контекст

В текущем ETL-потоке секреты проходят через доменные порты:
- на стадии `enrich` секретные значения выделяются в `secret_candidates`;
- далее `enrich` пишет их через `SecretStoreProtocol`;
- `resolve/plan` фиксируют `secret_fields` для операции;
- на `apply` адаптер гидрирует недостающие секреты через `SecretProviderProtocol`.

Этот поток как концепция корректен и соответствует разделению ответственности между стадиями.

Проблема находится не в идее потока, а в реализации хранилища и криптографии: сейчас используется временный dev-backend (`FileVault*`) в CSV, без шифрования, без lifecycle ключей и без production-grade error model.

---

## ⚠️ Проблема

Секретный контур не готов к production по безопасности и расширяемости:

- секреты хранятся в plaintext CSV;
- отсутствует архитектурно выделенный слой криптографии;
- отсутствует lifecycle ключей (источник, ротация, смена, аудит);
- отсутствует формализованный контракт для ошибок дешифрования/повреждения данных;
- storage-реализация и секретный контур развиваются как временный путь, а не как устойчивая подсистема.

Дополнительно зафиксирована policy-неопределённость на apply: если поле не входит в `secret_fields` для операции, поведение payload должно быть явно определено (для текущих решений: поле не отправляется).

---

## 🔍 Симптомы

- **Симптом 1**: `vault.csv` содержит реальные значения секретов в колонке `value`.
- **Симптом 2**: в проекте отсутствует реализация шифрования/дешифрования (`Fernet`/`cryptography` не интегрированы в секретный контур).
- **Симптом 3**: нет стандартизованных доменных ошибок для случаев `wrong key / corrupted ciphertext`.
- **Симптом 4**: ротация ключей не поддерживается и не зафиксирована как процесс.
- **Симптом 5**: fallback `vault -> prompt` в runtime не подходит для production/CI сценариев (интерактивность вместо детерминированного поведения).

---

## 📊 Масштаб проблемы

- **Частота**: Всегда при использовании текущего `FileVault*` пути.
- **Критичность**: Высокая.
- **Затронуто**: `import plan`, `enrich`, `import apply`, все датасеты, где используются секретные поля.

---

## 🧪 Как воспроизвести

1. Выполнить `import plan` с `--vault-file`.
2. Открыть созданный `vault.csv`.
3. **Ожидаемый результат**: секреты хранятся только в зашифрованном виде, ключи и crypto-policy управляются отдельно.
4. **Фактический результат**: секреты лежат в plaintext.

Дополнительно по policy apply:
1. Сформировать план, где для операции поле отсутствует в `secret_fields`.
2. Выполнить `import apply`.
3. **Ожидаемый результат**: секретное поле не попадает в payload этой операции.
4. **Фактический результат**: поведение зависит от dataset payload-контракта и требует явной фиксации.

---

## 🚫 Почему это проблема?

- Нарушаются базовые требования безопасности к хранению секретов.
- Невозможно безопасно управлять ключами при деплое и эксплуатации.
- Нет готовности к смене backend хранилища без переписывания слоя секретов.
- Нельзя гарантировать предсказуемое поведение в production при crypto-сбоях.
- Растёт риск утечек при эксплуатации, отладке и расширении подсистемы.

---

## 💡 Возможные решения (обсуждение)

### Вариант 1: Оставить CSV и добавить минимальное шифрование inline
- **Идея**: зашифровать `value` в существующем CSV backend без переразметки архитектуры.
- **Плюсы**: минимальный объём изменений.
- **Минусы**: слабая масштабируемость, неудобная ротация, хрупкий формат хранения, смешение crypto/storage ответственности.

### Вариант 2: Прямое шифрование в SQLite одним мастер-ключом
- **Идея**: хранить ciphertext в SQLite, ключ брать из ENV, без envelope-схемы.
- **Плюсы**: лучше операционно, чем CSV; проще, чем внешний vault.
- **Минусы**: смена мастер-ключа требует перешифровывать все записи.

### Вариант 3: Envelope encryption + hexagonal ports
- **Идея**: отдельные порты для storage и crypto; DEK хранится зашифрованным мастер-ключом; данные шифруются DEK.
- **Плюсы**: чёткое разделение ответственности, расширяемость backend, безопасная ротация мастер-ключа (через re-wrap DEK).
- **Минусы**: больше стартовой реализации и тестов.

### Вариант 4: Внешний Vault/KMS как единственный backend
- **Идея**: сразу интегрироваться с внешним хранилищем секретов.
- **Плюсы**: высокий security baseline.
- **Минусы**: существенно увеличивает сложность интеграции и операционный барьер на текущем этапе.
