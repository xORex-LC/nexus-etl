# TARGET-PROBLEM-004: Поведенческая спецификация провайдера захардкожена в Python

> **Статус**: Решена в [TARGET-DEC-004](./TARGET-DEC-004-target-dsl-declarative-provider.md)
> **Дата создания**: 2026-02-17
> **Затронутые компоненты**: `build_ankey_spec()`, `TargetSpec`, `AnkeyTargetProvider`, `connector/infra/target/providers/ankey_rest/spec.py`

---

## 📋 Контекст

TARGET-DEC-003 установил архитектуру TargetCore: `TargetRuntime` как facade,
`TargetSpec` как декларативная поведенческая спецификация провайдера (capabilities,
fault_rules, retry_rules, redaction, operations). ETL-пайплайн уже конфигурируется
декларативно через YAML (`datasets/*.yaml`) для всех стадий transform-слоя.

Для Ankey IDM вся поведенческая спецификация сосредоточена в `build_ankey_spec()` —
Python-функции, конструирующей `TargetSpec` вручную.

---

## ⚠️ Проблема

Поведенческая спецификация target-провайдера жёстко зашита в Python-код. Отсутствует
механизм декларативной конфигурации, симметричный transform-DSL:

- Добавление нового провайдера = написание `build_*_spec()` в Python + регистрация.
- Изменение fault/retry-правил требует правки кода + коммит + деплой.
- `TargetSpec` содержит все необходимые структуры (Pydantic, frozen, validators),
  но нет пути загрузить её из внешнего конфигурационного источника.
- Нарушается принцип «конфигурация отделена от кода», уже принятый в проекте.

---

## 🔍 Симптомы

- **Симптом 1**: добавление операции `users.delete` требует правки `spec.py` → новый коммит → деплой.
- **Симптом 2**: нет способа переопределить `retry_config.max_attempts` через конфиг среды без правки кода.
- **Симптом 3**: добавление второго HTTP-провайдера (например, Keycloak) требует дублирования структуры `build_*_spec()` вместо описания YAML-файла.
- **Симптом 4**: spec и wiring-код живут в одном пакете без разделения ответственности.

---

## 📊 Масштаб проблемы

- **Частота**: Всегда (каждый новый провайдер или изменение spec требует Python-кода)
- **Критичность**: Средняя (система работает, но нарушает принцип extensibility)
- **Затронуто**: `AnkeyTargetProvider`, будущие HTTP/SQL/file провайдеры, CI/CD процесс

---

## 🧪 Как воспроизвести

1. Поставить задачу: добавить второй HTTP-провайдер `keycloak_rest`.
2. Попытаться описать его spec декларативно, не трогая Python.
3. **Ожидаемый результат**: YAML-файл + регистрация провайдера — достаточно.
4. **Фактический результат**: необходимо написать `build_keycloak_spec()` в Python.

---

## 🚫 Почему это проблема?

- Нарушает принцип «конфигурация отделена от кода», уже принятый для transform-слоя.
- Повышает порог входа при добавлении провайдеров и изменении fault/retry-политики.
- Исключает возможность конфигурирования через CI/CD без деплоя кода.
- Нет единообразия: transform-стадии декларативны, target-провайдер — нет.

---

## 💡 Возможные решения (обсуждение)

### Вариант 1: Расширить `dsl/specs/` и `dsl/loader/`
- **Идея**: добавить `specs/target.py` и `loader/target.py` внутрь существующего `dsl` пакета.
- **Плюсы**: максимальное переиспользование кода без нового пакета.
- **Минусы**: смешивает transform-DSL (правила обработки данных) и infrastructure-DSL
  (описание внешнего API) в одном модуле — нарушает SRP.

### Вариант 2: Отдельный `target_dsl` модуль, использующий утилиты dsl-core
- **Идея**: `connector/domain/target_dsl/` — самостоятельный модуль, импортирующий
  generic-утилиты из dsl-core (DslLoadError, loader utilities).
- **Плюсы**: чёткое разделение, переиспользование утилит без смешения концернов.
- **Минусы**: требует минимального расширения публичного API dsl-core.
