# CONFIG-PROBLEM-002: Ручная валидация Settings и отсутствие Pydantic в конфиг-слое

> **Статус**: Решена в CONFIG-DEC-002
> **Дата создания**: 2026-02-19
> **Затронутые компоненты**: `connector/config/config.py`, `connector/config/app_settings.py`

---

## 📋 Контекст

Settings-слой был реализован на основе замороженных dataclass'ов с ручной валидацией
(`_validate_settings()`). На момент первоначальной реализации это было обосновано простотой.
Однако с ростом проекта количество полей и правил валидации значительно увеличилось.
При этом Pydantic уже используется в других частях проекта.

---

## ⚠️ Проблема

Settings-слой реализует вручную то, что Pydantic предоставляет декларативно:

1. **Ручная валидация** — `_validate_settings()` накапливает ошибки в список, проверяет
   enum-значения через `if value not in {"A", "B", "C"}`, range-проверки через `if not (lo <= x <= hi)`
2. **Ручная коэрция типов** — ENV-переменные приходят как `str`; код вручную конвертирует
   `"5000"` → `int`, `"true"` → `bool` и т.д.
3. **Дублирование** — каждое новое поле требует: добавить в `Settings`, добавить валидацию,
   добавить в `_SLICE_FIELD_MAP`, добавить в slice, добавить билдер-логику
4. **Слабая типизация** — `str` вместо `Literal["WAL", "DELETE", ...]`, нет range-аннотаций
5. **Нет схемы для документации** — невозможно автоматически сгенерировать список всех
   параметров с типами и дефолтами

---

## 🔍 Симптомы

- **Симптом 1**: Добавление нового enum-поля (например, `sqlite_journal_mode`) требует
  написать несколько строк ручной проверки в `_validate_settings()` вместо `Literal[...]`
- **Симптом 2**: Range-валидация (`busy_timeout_ms: 1000–30000`) — отдельный `if` в
  `_validate_settings()`, а не `Field(ge=1000, le=30000)`
- **Симптом 3**: Ошибки конфигурации обнаруживаются в runtime (при запуске приложения),
  а не при загрузке модуля (Pydantic валидирует при создании модели)
- **Симптом 4**: IDE не подсказывает допустимые значения для enum-полей — нет `Literal`

---

## 📊 Масштаб проблемы

- **Частота**: При каждом добавлении нового конфигурируемого параметра
- **Критичность**: Средняя (работает, но высокая стоимость сопровождения)
- **Затронуто**: Все команды, все settings-слайсы, все новые параметры (включая `SqliteSettings`)

---

## 🧪 Как воспроизвести

1. Открыть `connector/config/config.py`, найти `_validate_settings()`
2. Посчитать строки ручных проверок для уже существующих полей
3. Попробовать добавить `sqlite_journal_mode: Literal["WAL", "DELETE", "MEMORY", ...]`
4. **Ожидаемый**: добавить поле с типом `Literal` — Pydantic валидирует автоматически
5. **Фактический**: добавить поле в `Settings` (str) + добавить проверку в `_validate_settings()`

---

## 🚫 Почему это проблема?

- Ручная валидация отстаёт от типов: поле может быть `str`, но проверяется как enum —
  рассинхронизация между типом и логикой
- Стоимость сопровождения растёт: чем больше параметров, тем длиннее `_validate_settings()`
- Pydantic уже в проекте — отсутствие его использования здесь выглядит как исключение,
  а не как правило

---

## 💡 Возможные решения (обсуждение)

### Вариант 1: Частичный рефакторинг — Literal + Field для новых полей

- **Идея**: Оставить dataclass, но добавить проверки через `__post_init__` или отдельный валидатор
- **Плюсы**: Минимальные изменения
- **Минусы**: Не решает коэрцию типов, не устраняет ручной код, не унифицирует подход

### Вариант 2: Pydantic BaseSettings + pydantic-settings

- **Идея**: `Settings` наследует `BaseSettings`, слайсы — `BaseModel`; `pydantic-settings`
  обрабатывает env vars, типы, валидацию
- **Плюсы**: Декларативно, автокоэрция, `Literal` и `Field` работают из коробки,
  возможна автогенерация документации
- **Минусы**: Требует миграции существующей логики загрузки (приоритет CLI > ENV > file > defaults)
