# DSL-PROBLEM-002: Архитектурная связность DSL Core и дрейф контрактов при росте

> **Статус**: Решена в [DSL-DEC-002](./DSL-DEC-002-modular-dsl-core-and-contract-stabilization.md)
> **Дата создания**: 2026-02-13
> **Затронутые компоненты**: `connector/domain/dsl/specs/*`, `connector/domain/dsl/loader/*`, `connector/domain/dsl/build_options.py`, `connector/domain/dsl/ops.py`, `connector/infra/cache/sync/dsl_adapter.py`

---

## 📋 Контекст

После закрытия fail-fast проблем из `DSL-PROBLEM-001` DSL Core оставался корректным функционально, но начал упираться в архитектурный рост.  
Архитектурный аудит DSL Core зафиксировал, что наибольший риск сместился из correctness в maintainability/scalability:
- высокая связность и перегруженные модули;
- неоднозначные compile/runtime контракты в отдельных ветках;
- избыточная стоимость hot-path операций при больших объёмах.

Целью было вернуть предсказуемость архитектуры DSL Core при росте числа датасетов и правил, без отката от уже принятых fail-fast принципов.

---

## ⚠️ Проблема

DSL Core развивался как единый «толстый» слой, где модели, загрузка, merge-политики и часть runtime-семантики были связаны слишком плотно.  
Это приводило к дрейфу контрактов между transform/cache путями и ухудшало масштабируемость:
- изменения в одной зоне затрагивали нецелевые зоны;
- часть конфликтов в build options могла скрываться до runtime;
- поведение при ошибках было неоднородным между доменами;
- производительность ряда DSL ops деградировала на массовых данных.

---

## 🔍 Симптомы

- **Симптом 1**: высокая сложность сопровождения из-за монолитной структуры `specs`/`loader`.
- **Симптом 2**: неочевидные конфликты cache runtime build options при нескольких dataset overrides.
- **Симптом 3**: хрупкая привязка резолва корня проекта к фиксированной глубине пакета.
- **Симптом 4**: runtime стоимость regex/mapping операций росла пропорционально количеству строк.
- **Симптом 5**: неоднородность error-handling vocabulary и runtime обработки ошибок между transform/cache.

---

## 📊 Масштаб проблемы

- **Частота**: Иногда (проявляется при росте количества датасетов/правил и при cross-domain изменениях).
- **Критичность**: Высокая (замедляет эволюцию DSL Core и повышает риск скрытых регрессий).
- **Затронуто**: DSL maintainers, stage DSL компиляторы, cache sync path, команды, добавляющие новые датасеты.

---

## 🧪 Как воспроизвести

1. В `registry.yml` задать `cache.datasets` с `build_options.cache` одновременно для нескольких датасетов.
2. Вызвать `load_cache_build_options_for_runtime()` без явного `dataset_overrides`.
3. Запустить runtime path, зависящий от этих опций.
4. **Ожидаемый результат**: либо явный детерминированный источник overrides, либо fail-fast ошибка о неоднозначности.
5. **Фактический результат (до решения)**: возникал риск «последний wins» без явного контракта источника.

Дополнительно:
1. Выполнить массовый прогон `extract_patterns`/`map_dict(casefold=True)` на большом объёме строк.
2. **Ожидаемый результат**: переиспользование вычислений и стабильный throughput.
3. **Фактический результат (до решения)**: повторная компиляция/нормализация на hot-path.

---

## 🚫 Почему это проблема?

- Нарушает архитектурную модульность DSL Core (сложно безопасно расширять).
- Повышает вероятность неявных конфигурационных конфликтов в runtime.
- Увеличивает стоимость изменений и ревью при росте DSL.
- Создаёт performance-риск для ETL-сценариев с большим потоком данных.
- Ухудшает предсказуемость operational behavior при ошибках.

## 💡 Возможные решения (обсуждение)

> Этот раздел может содержать первоначальные идеи до принятия финального решения

### Вариант 1: Локальные точечные фиксы без декомпозиции
- **Идея**: оставить текущую структуру и чинить только конкретные дефекты.
- **Плюсы**: минимальный порог изменений в краткосрочной перспективе.
- **Минусы**: не устраняет root cause высокой связности и дрейфа контрактов.

### Вариант 2: Модульная декомпозиция + нормализация контрактов
- **Идея**: разделить DSL Core по доменным зонам (`specs/*`, `loader/*`) и формализовать strict/merge/error semantics.
- **Плюсы**: лучшее разделение ответственности, предсказуемые контракты, проще масштабирование.
- **Минусы**: рефакторинг затрагивает несколько компонентов и требует обновления документации/тестов.

### Вариант 3: Частичная декомпозиция только docs/API
- **Идея**: оставить кодовую структуру, но документировать правила и ограничения.
- **Плюсы**: быстро, без риска миграции кода.
- **Минусы**: не решает техническую связность и runtime-риски.
