# DSL-PROBLEM-001: Fail-late поведение и слабый compile-контракт в DSL Core

> **Статус**: Решена в [DSL-DEC-001](./DSL-DEC-001-strict-compile-validation-and-diagnostics-hardening.md)
> **Дата создания**: 2026-02-12
> **Затронутые компоненты**: `specs.py`, `loader.py`, `build_options.py`, `engine.py`, stage DSL компиляторы

---

## 📋 Контекст

`DSL Core` стал единым слоем для нескольких стадий (`mapping`, `normalize`, `enrich`, cache DSL compile path).  
На фоне роста числа декларативных YAML-спеков появились системные ошибки качества compile/load-контракта:
1. Часть конфигурационных ошибок не выявлялась на этапе сборки.
2. Ошибки смещались в runtime и проявлялись уже на потоке данных.
3. Диагностика runtime-падений была недостаточно информативной.

Целевой архитектурный принцип DSL-слоя: **максимум ошибок должен ловиться на load/compile этапе (fail-fast)**, до запуска стадий трансформации.

---

## ⚠️ Проблема

**Суть**: в DSL Core наблюдалась fail-late модель, при которой критичные ошибки конфигурации и компиляции не блокировали выполнение заранее.

Ключевые дефекты:
1. Неизвестные ключи в DSL-моделях могли быть пропущены.
2. Неизвестный dataset в build options не приводил к жёсткой ошибке.
3. Часть compile-policy флагов была объявлена, но не влияла на поведение.
4. Неизвестные DSL-операции в ряде путей выявлялись только при runtime apply.
5. `DSL_OP_FAILED` не содержал полного контекста шага/операции.

**Технически**:
- compile/load и execution path были несогласованы по strictness;
- часть invariant-checks выполнялась поздно в `TransformationEngine.apply`, а не в compile;
- diagnostics payload не позволял быстро локализовать конкретный `op`/`args`/`step`.

---

## 🔍 Симптомы

- **Симптом 1**: опечатка в YAML не всегда падала на compile этапе.
- **Симптом 2**: неверный dataset-id мог тихо уйти в defaults.
- **Симптом 3**: `strict` и `fail_on_unknown_ops` давали неполный защитный эффект.
- **Симптом 4**: runtime ошибки DSL-операций требовали ручной локализации по логам и коду.

Пример аномалии:
1. Конфиг содержит неизвестный/ошибочный оператор.
2. Компиляция стадии проходит.
3. Ошибка возникает только на конкретной записи во время `TransformationEngine.apply`.

Пример из runtime-диагностики (до исправлений):
```text
code=DSL_OP_FAILED
message="Operation failed: ..."
details={"error": "..."}
```
^ Недостаточно данных: нет полного контекста шага и параметров операции.

---

## 📊 Масштаб проблемы

- **Частота**: Иногда (зависит от качества конкретной спек-файла и wiring)
- **Критичность**: Высокая (архитектурный риск fail-late в базовом DSL-ядре)
- **Затронуто**:
  - DSL compile/load pipeline,
  - все стадии, использующие единый DSL слой,
  - наблюдаемость runtime диагностики DSL.

---

## 🧪 Как воспроизвести

1. Добавить в stage DSL YAML неизвестный ключ или неизвестную операцию.
2. Запустить compile/load путь для соответствующей стадии.
3. Запустить выполнение стадии на данных.
4. **Ожидаемый результат**: ошибка должна быть остановлена на load/compile.
5. **Фактический результат (до исправлений)**: часть ошибок проявлялась только в runtime.

Минимальный сценарий:
1. В `datasets/*.yaml` добавить неизвестный `op`.
2. Выполнить сборку стадии.
3. Пропустить через стадию строку данных.
4. Наблюдать падение в execution path вместо compile-time fail-fast.

---

## 🚫 Почему это проблема?

1. Нарушает контракт fail-fast для конфигурационного слоя.
2. Увеличивает риск продакшен-регрессий при обновлении DSL-правил.
3. Усложняет добавление новых датасетов и сопровождение существующих.
4. Повышает стоимость диагностики и время восстановления.

---

## 💡 Возможные решения (обсуждение)

### Вариант 1: Оставить мягкое поведение (как есть)
- **Идея**: не ужесточать compile/load слой, компенсировать runtime-тестами.
- **Плюсы**: минимальные изменения, отсутствие миграционных падений старых конфигов.
- **Минусы**: сохраняется fail-late и системный архитектурный риск.

### Вариант 2: Полный fail-fast контракт в DSL Core
- **Идея**: ужесточить pydantic модели, loader, compile-policy и контекст диагностик.
- **Плюсы**:
  - ошибки ловятся раньше,
  - предсказуемое поведение strict policy,
  - лучшее качество observability.
- **Минусы**:
  - потребуется чистка старых/неточных YAML-конфигов,
  - более строгий режим может выявить накопленный техдолг сразу.

### Вариант 3: Ужесточать только отдельные стадии
- **Идея**: точечно исправлять mapper/normalizer/enricher/cache независимо.
- **Плюсы**: можно внедрять постепенно, с меньшим объёмом изменения за шаг.
- **Минусы**:
  - сохраняется несогласованность compile-контрактов,
  - дублируются политики и правила валидации,
  - архитектурный долг переносится между стадиями.
