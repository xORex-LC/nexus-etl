### Dataclass: `ModelName`

**Назначение**: Что представляет эта модель

**Структура**:
```python
@dataclass(frozen=True)
class ModelName:
    field1: str                    # Описание поля
    field2: int | None             # Опциональное поле
    field3: dict[str, Any]         # Словарь данных
    field4: list[str] = field(default_factory=list)  # Поле с дефолтом
```

**Создание и использование**:
```python
# Как создаётся
instance = ModelName(
    field1="value",
    field2=42,
    field3={"key": "value"}
)

# Где используется
def process(data: ModelName) -> Result:
    # Использование модели
    ...
```

**Lifecycle**:
1. **Создание**: Где и когда создаётся (например, в методе `transform()` после валидации)
2. **Трансформации**: Как изменяется (если immutable - указать, что frozen=True)
3. **Завершение**: Куда передаётся дальше (например, в следующую стадию pipeline)

**Инварианты**:
- Инвариант 1: Условие, которое всегда должно выполняться для полей
- Инвариант 2: Ограничения значений

**Расположение**: `connector/path/to/file.py:LINE_NUMBER`

**Сигнатура**:
```python
def method_name(
    self,
    param1: Type1,
    *,
    param2: Type2,
    param3: Type3 | None = None,
) -> tuple[Result, list[Error], list[Warning]]:
    """
    Docstring из кода (если есть).
    """
```

**Назначение**:
[1-2 предложения - что делает метод, какую задачу решает]

---

**Алгоритм** (выбрать формат):

*Вариант A: Pseudocode с номерами строк*
```
1. Validation (lines X-Y)
   - Check param1 is not empty
   - Ensure param2 meets constraints
   - IF invalid → return error

2. Main processing (lines Y-Z)
   - FOR EACH item IN collection:
       → IF condition1: apply transformation A
       → ELSE IF condition2: apply transformation B
       → ELSE: handle error case

3. Build result (lines Z-W)
   - Aggregate processed items
   - Construct return tuple

4. Cleanup (lines W-V)
   - Finalize resources
```

*Вариант B: ASCII Flow диаграмма* (для более наглядных случаев)
```
Input
  ↓
[Validation] → {Valid?} ─No→ Error Exit
  ↓ Yes
[Step 1: Transform]
  ↓
[Step 2: Decision Point]
  ├─ Case A → [Handle A]
  ├─ Case B → [Handle B] ──→ [Retry Loop] ──┐
  └─ Case C → [Handle C]                     │
                ↓                            │
          [Aggregate Results] ←──────────────┘
                ↓
             Output
```

---

**Временная сложность**:
- **Best case**: O(1) - когда [условие, например: все данные валидны с первой попытки]
- **Average case**: O(n log n) - типичный сценарий [объяснение]
- **Worst case**: O(n²) - когда [условие, например: максимальное количество конфликтов]

*Если есть зависимость от размера данных, определить переменные*:
- n = количество записей в коллекции
- k = количество ключей для lookup
- m = количество правил deduplication

---

**Инварианты**:
1. **Инвариант 1**: Всегда возвращает tuple из 3 элементов
2. **Инвариант 2**: Не мутирует входные параметры (работает с копиями)
3. **Инвариант 3**: Errors и warnings не пересекаются по записям

**Edge cases**:
1. **Пустой input**: Возвращает пустой результат без ошибок
2. **Partial data**: Обрабатывает неполные данные согласно параметру allow_partial
3. **Timeout scenario**: Прерывает обработку после N попыток
4. **Conflict resolution**: Применяет стратегию из conflict_policy

**Связанные методы**:
- `_helper_method()` line XXX - вспомогательная функция для шага 2
- `OtherClass.validate()` - валидация используемая в шаге 1

**Performance заметки** (опционально):
- **Bottleneck**: Описание узкого места (например, N запросов к БД)
- **Optimization**: Текущая оптимизация (например, batch processing)
- **Benchmark**: Данные производительности, если есть (10K записей: ~5 сек)

## 🔌 Контракты и границы

> **Назначение**: Явное описание контрактов между слоями и архитектурных границ

### DSL-контракт (если применимо)

**Входной формат** (YAML/JSON):
```yaml
# Пример декларативной конфигурации
resolve:
  rules:
    - field: employee_id
      strategy: prefer_source
      resolve_keys: ["id", "source_id"]
```

**Схема валидации**: `connector/domain/dsl/schemas/[layer]_schema.json`

**Обязательные поля**: `field1`, `field2`

**Опциональные поля**: `field3`, `field4`

**Пример невалидной конфигурации**:
```yaml
# ❌ Ошибка: отсутствует обязательное поле 'field1'
resolve:
  rules:
    - field2: value  # Неполная конфигурация
```

---

### Runtime-контракт

**Что получает инфраструктурный слой после компиляции DSL** (если применимо):

```python
@dataclass(frozen=True)
class CompiledRules:
    rules: dict[str, callable]     # field_name → strategy function
    metadata: dict[str, Any]        # Дополнительные метаданные
```

**Гарантии**:
- Все правила прошли валидацию DSL Loader
- Функции callable и соответствуют ожидаемой сигнатуре
- No runtime compilation (все скомпилировано заранее)

**Используется в**: `CoreClass.__init__(compiled_rules=...)`

**Пример использования**:
```python
# После компиляции DSL
rules = dsl.compile(yaml_config)
# Передаётся в Core
core = CoreClass(rules=rules)
```

### Границы слоёв

**Разрешенные зависимости** (что можно импортировать):
- ✅ `CoreClass` → `PortProtocol` (Protocol) — абстракция для внешних зависимостей
- ✅ `CoreClass` → `SharedModels` — общие доменные модели (dataclass'ы)
- ✅ `DslCompiler` → `CoreClass` — компиляция в core logic
- ✅ `ServiceLayer` → `CoreClass` — использование бизнес-логики

**Запрещенные зависимости** (нарушение архитектуры):
- ❌ `CoreClass` → `connector/infra/*` — нарушение Ports & Adapters
- ❌ `DslCompiler` → `Specific Adapters` — DSL должен быть infrastructure-agnostic
- ❌ `CoreClass` → `UseCase` — Core не знает о use cases (обратная зависимость)
- ❌ `CoreClass` → `FastAPI`, `SQLAlchemy`, etc. — Core не зависит от фреймворков

**Архитектурные тесты** (если есть): `tests/architecture/test_layer_boundaries.py`

**Визуальная граница**:
```
┌─────────────────────────────────────────┐
│ Infrastructure (Adapters)               │  ← Реализация портов, внешние системы
└────────────▲────────────────────────────┘
             │ implements Ports
┌────────────┴────────────────────────────┐
│ Domain (Core + DSL)                     │  ← Бизнес-логика, не знает про инфру
└────────────▲────────────────────────────┘
             │ uses
┌────────────┴────────────────────────────┐
│ Application (UseCases)                  │  ← Оркестрация workflow
└─────────────────────────────────────────┘
```

### Взаимодействие с доменными слоями

| Слой | Направление | Через что | Контракт | Пример |
|------|------------|-----------|----------|--------|
| Cache Core | Зависимость | `RuntimePort` (Protocol) | `lookup()`, `create_pending()` | `ResolveCore` использует cache для FK resolution |
| Transform Core | Вызывает | Direct call | `InputModel` → `OutputModel` | `MatchCore` → `ResolveCore` |
| Mapping DSL | Shared Models | Dataclasses | `Identity`, `RowRef` | Общие модели передаются между слоями |

**Важно**:
- Все зависимости должны идти **сверху вниз** (от Application к Domain к Infrastructure)
- Domain **никогда** не зависит от Infrastructure напрямую (только через Ports)
- Shared Models могут использоваться на всех уровнях (но определены в Domain)

### 🚨 Failure Modes

> **Назначение**: Документация ошибок и исключений, которые могут возникать в слое

| Исключение | Условие возникновения | Поведение системы | Как обработать |
|------------|----------------------|-------------------|---------------|
| `CyclicDependencyError` | Граф зависимостей содержит цикл | ValueError в `_topological_order()` line 112, fail-fast | Исправить конфигурацию `cache.yaml`, убрать циклические зависимости |
| `AmbiguousMatchError` | Dedup нашёл несколько кандидатов | Возвращает None + error в `resolve()` | Уточнить dedup_rules или fingerprint_fields в конфигурации |
| `ImmutableFieldMutationError` | Попытка изменить immutable поле sink | Sink validation блокирует, error на line 185 | Проверить sink_spec, убрать поле из desired_state |
| `MissingDependencyError` | Отсутствует обязательная зависимость (Port) | ValueError при инициализации `__init__` | Передать все required dependencies через конструктор |

**Формат записи**:
- **Исключение**: Имя exception class (если есть) или общее описание ошибки
- **Условие**: При каких входных данных / состоянии возникает
- **Поведение**: Что делает система (raise exception, return None + error, fallback)
- **Как обработать**: Конкретные действия для исправления (изменить конфигурацию, исправить данные, etc)

**Примеры типичных failure modes**:
- Validation failures (некорректные входные данные)
- Configuration errors (неправильная конфигурация DSL)
- Dependency failures (недоступен внешний сервис через Port)
- Business rule violations (нарушение invariant'ов)
- Resource exhaustion (timeout, memory limit)

**Связь с monitoring**:
- Все failure modes должны логироваться с соответствующим уровнем (ERROR, WARNING)
- Критические errors должны триггерить алерты

### ⏱️ Performance заметки

**Узкие места**:
1. **Операция X** (`method_name` line 150)
   - **Проблема**: O(n²) при большом объёме данных
   - **Текущая оптимизация**: Batch processing по 1000 записей
   - **Дальнейшие планы**: Индексирование в памяти для ускорения lookup

2. **Операция Y** (`another_method` line 250)
   - **Проблема**: N запросов к БД для каждого элемента
   - **Текущая оптимизация**: Pre-loading в batch_index
   - **Эффект**: Сокращение с O(n) запросов до O(1)

**Оптимизации**:
- **Caching**: Кэш lookup-ключей на время обработки батча (экономит N запросов к БД)
- **Lazy evaluation**: Использование генераторов вместо списков для stream processing
- **Batch operations**: Группировка операций по 1000 записей для снижения overhead

**Benchmark данные** (если есть):
- Обработка 10K записей: ~5 сек
- Обработка 100K записей: ~45 сек (не линейно из-за GC)
- Memory peak: ~500MB для 100K записей
