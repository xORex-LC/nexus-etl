# Resolver: Полная карта функциональных возможностей и аудит ответственности

## Кратко

Этот документ описывает фактическое поведение стадии разрешения операций и ссылок на основе кода и тестов: как она принимает решение о дальнейшей операции, как разрешает ссылочные значения, как работает с отложенными связями, какие побочные эффекты создаёт и где проходит граница между логикой данных и технической/переходной обвязкой.

Основной текст написан в терминах бизнес-логики и алгоритмов без опоры на кодовый нейминг. Точные ссылки на реализацию вынесены в раздел трассировки.

## 1. Назначение стадии и границы ответственности

### Что стадия делает

Стадия разрешения получает результат сопоставления и превращает его в план действия для последующего применения.

Ключевые задачи стадии:

1. Проверить, можно ли продолжать обработку после результата сопоставления.
2. Подготовить финальное желаемое состояние записи (включая merge/fill-from-existing, если настроено).
3. Разрешить ссылочные поля через lookup в индексах/кэше и внутри текущего батча.
4. При невозможности разрешения ссылки создать отложенную задачу (pending) или завершить ошибкой — по политике.
5. Вычислить целевую операцию (создать/обновить/пропустить).
6. Подготовить структуру результата для следующей стадии применения.

### Что стадия не должна делать (архитектурная граница)

1. Не должна напрямую выполнять изменения в целевой системе.
2. Не должна реализовывать инфраструктуру хранения индексов и pending-записей.
3. Не должна содержать transport/delivery-логику CLI/API.
4. Не должна быть единственным владельцем сериализации/десериализации storage-представлений (это спорная зона и фактически частично нарушено).

### Фактические расширения ответственности (есть в реализации)

1. Стадия частично занимается housekeeping pending-записей (периодический sweep просроченных).
2. Стадия выполняет сериализацию pending payload в JSON.
3. Стадия запускает проверку схемы sink для изменённых полей.
4. В пакете стадии присутствует отдельный декодер pending payload обратно в типизированный результат.

Эти зоны разобраны в аудите ответственности ниже.

## 2. Что стадия получает на вход

### На уровне одной записи

Стадия получает результат сопоставления одной строки и дополнительные вспомогательные данные для разрешения ссылок и выбора операции.

Вход включает:

1. Результат сопоставления (идентичность, статус сопоставления, желаемое состояние, найденная целевая запись, fingerprint, explainability-данные).
2. Словарь заранее известных целевых идентификаторов для текущего прогона (по ключу идентичности).
3. Дополнительные метаданные строки (в т.ч. переопределения lookup-ключей для ссылочных полей).
4. Опциональный индекс текущего батча для ускоренного разрешения ссылок между записями одного прогона.

### На уровне конфигурации/окружения

Стадия использует:

1. Правила сборки и сравнения данных (merge, diff, source reference, секретные поля, lifecycle секретов).
2. Правила разрешения ссылок (какие поля являются ссылками, по каким ключам искать, как сужать кандидатов, что делать при нерешённой ссылке, coercion типа).
3. Порт runtime-доступа к identity/pending-хранилищу.
4. Runtime-настройки pending-механики (TTL, max attempts, интервал sweep, partial policy и др.).
5. Каталог диагностик.
6. Опциональную схему целевой стороны для проверки типов изменённых полей.

### Важные предпосылки входа

1. Результат сопоставления может быть неоднозначным/конфликтным — стадия должна это остановить.
2. Runtime-порт может отсутствовать; в этом случае часть функций стадии недоступна.
3. Ссылочные поля могут быть уже в финальном типе (например, числовой идентификатор), и повторный lookup не всегда нужен.
4. Метаданные могут содержать переопределения ключей для ссылочного lookup, которые имеют приоритет над значениями из состояния.

## 3. Что стадия возвращает на выход

### Основной контракт вызова

Для каждой входной строки стадия возвращает тройку:

1. Результат разрешения (или `None`, если обработка остановлена/отложена).
2. Список ошибок.
3. Список предупреждений.

### Возможные варианты результата

1. **Готовый план действия**
   - содержит финальное желаемое состояние (с учётом resolved links);
   - содержит тип операции (создать/обновить/пропустить);
   - содержит diff для обновления (если нужен);
   - содержит целевой идентификатор;
   - содержит source reference (если настроено);
   - содержит список secret-полей и политику lifecycle секретов.

2. **Нет результата, но есть предупреждение о pending**
   - возникает при нерешённой ссылке и политике остановки без partial;
   - pending-запись создана;
   - строка должна быть обработана позже.

3. **Есть результат и предупреждение о pending**
   - возникает при нерешённой ссылке и разрешённом partial-режиме;
   - план действия формируется, но часть ссылок остаётся в исходном виде до последующего разрешения.

4. **Нет результата из-за ошибок**
   - конфликт/неоднозначность после стадии сопоставления;
   - нерешённая ссылка с hard-error политикой;
   - превышен лимит попыток pending;
   - отсутствует целевой идентификатор;
   - ошибка проверки схемы изменённых полей;
   - конфигурационная проблема (например, ссылка требует runtime-порт, а он не настроен).

### Дополнительный side-channel результата стадии

Стадия накапливает сведения о просроченных pending-записях, обнаруженных при периодическом sweep. Эти записи можно получить отдельным вызовом "drain"-метода (с очисткой внутреннего буфера).

## 4. Состояние, которое стадия хранит/использует во время прогона

### Локальное состояние экземпляра

1. Правила сборки/сравнения/секретов.
2. Правила разрешения ссылочных полей.
3. Ссылка на runtime-порт identity/pending.
4. Runtime-настройки pending-механики.
5. Каталог диагностик.
6. Схема целевой стороны (опционально).
7. Время последнего sweep просроченных pending.
8. Буфер обнаруженных просроченных pending-записей (до чтения вызывающей стороной).

### Состояние, читаемое/пишущееся через порт

1. Поиск кандидатов по identity-ключу.
2. Создание pending-записи для нерешённой ссылки.
3. Инкремент/чтение счётчика попыток обработки pending.
4. Пометка pending как конфликтной записи при превышении лимита попыток.
5. Пометка исходной строки как полностью resolved (если pending не создавался).
6. Sweep просроченных pending-записей.

### Особенности жизненного цикла состояния

1. Sweep выполняется не на каждом вызове, а по интервалу.
2. Буфер просроченных pending-записей растёт между вызовами и очищается только явным "drain".
3. Если runtime-порт не задан, housekeeping и lookup/pending функции отключаются, но часть логики стадии (без ссылочного resolve) может продолжать работать.

## 5. Полная карта функциональных возможностей

### 5.1 Предварительная проверка статуса сопоставления (гейт на входе)

Стадия не пытается строить план действия, если предыдущая стадия вернула конфликтный или неоднозначный результат.

Поддерживаемое поведение:

1. Жёсткая остановка при неоднозначности сопоставления.
2. Жёсткая остановка при конфликте внутри входного батча, зафиксированном предыдущей стадией.
3. Формирование диагностической ошибки стадии разрешения с привязкой к строке.

Это важная бизнес-политика: при неразрешённом выборе целевой записи план применения не строится.

### 5.2 Периодический housekeeping просроченных pending-связей

Перед обработкой строки стадия может запускать sweep просроченных отложенных связей.

Поддерживаемые возможности:

1. Полное отключение sweep при отсутствии runtime-порта.
2. Полное отключение sweep при нулевом/отрицательном интервале.
3. Выполнение sweep только если прошёл конфигурационный интервал с момента прошлого sweep.
4. Накопление результатов sweep во внутреннем буфере для последующего чтения вызывающей стороной.

Это не влияет на решение по текущей строке напрямую, но создаёт наблюдаемое побочное поведение стадии.

### 5.3 Применение merge/fill-from-existing к желаемому состоянию

Стадия может модифицировать желаемое состояние на основе уже существующей записи (например, заполнять пустые поля значениями из текущих данных).

Поддерживаемые возможности:

1. Вызывается только если merge-политика настроена.
2. Работает на копии входного состояния, не мутируя исходный результат предыдущей стадии.
3. Защищает явно заданные значения источника от перезаписи merge-политикой.
4. Если merge-политика пытается перезаписать явно заданные значения, стадия:
   - восстанавливает исходные значения;
   - пишет предупреждение в лог (не в диагностику строки).
5. Отслеживает поля, фактически изменённые merge-политикой, для последующей selective validation.

### 5.4 Разрешение ссылочных полей (lookup link resolution)

Стадия умеет преобразовывать ссылочные поля из исходных ключей/представлений в целевые идентификаторы.

#### 5.4.1 Выбор полей для обработки

Для каждого настроенного ссылочного поля стадия:

1. Пропускает поле, если оно отсутствует в текущем желаемом состоянии.
2. Пропускает поле, если значение `None`.
3. Может пропустить lookup, если значение уже является числом и правила не требуют поиска по целевому идентификатору (защита от повторного преобразования).

#### 5.4.2 Извлечение lookup-ключей

Стадия формирует lookup-набор значений для ссылочного поля из двух источников с приоритетом:

1. Переопределения из метаданных строки (если есть).
2. Значения из текущего желаемого состояния.

Поведение при формировании ключей:

1. `None` и пустые строки игнорируются.
2. Числа приводятся к строке.
3. Строки очищаются от внешних пробелов.

#### 5.4.3 Поиск кандидатов по ссылке

Для каждого ключа lookup (в заданном порядке) стадия:

1. Строит канонический identity-ключ.
2. Сначала проверяет индекс текущего батча (если передан).
3. Если в батче ничего не найдено, обращается к runtime-порту.
4. Если найден ровно один кандидат — ссылка считается разрешённой.
5. Если найдено несколько кандидатов — применяет дополнительные правила сужения (dedup rules).
6. Если после сужения остался один кандидат — ссылка разрешена.
7. Если кандидатов всё ещё несколько — ссылка считается неоднозначной.
8. Если кандидатов нет по текущему ключу — переходит к следующему ключу.

#### 5.4.4 Сужение множественных кандидатов по доп. правилам

Стадия поддерживает последовательное сужение списка кандидатов через набор правил, каждое из которых задаёт пересечение по дополнительным ключам.

Поддерживаемое поведение:

1. Сужение выполняется только если доп. правила настроены.
2. Каждое правило представляет набор ключей, по которым строится пересечение кандидатов.
3. Для каждого ключа внутри правила значение берётся:
   - из ранее сформированных lookup-значений;
   - если нет — из текущего желаемого состояния.
4. Если для любого ключа внутри правила значение отсутствует/пустое, это правило не может сузить кандидатов.
5. Если пересечение по правилу непустое, оно становится новым списком кандидатов.
6. Если после очередного правила остался один кандидат, сужение завершается досрочно.

Это позволяет разрешить ссылку в условиях неуникальности одного lookup-ключа.

#### 5.4.5 Преобразование типа resolved значения

После выбора единственного кандидата стадия может привести его к типу поля назначения (например, в число). Если преобразование не удалось, сохраняется исходное строковое значение идентификатора.

### 5.5 Поведение при нерешённой ссылке (pending/hard-error)

Если ссылка не разрешилась (нет кандидатов или осталась неоднозначность), стадия применяет политику для этого поля.

Поддерживаемые варианты:

1. **Жёсткая ошибка**
   - формируется ошибка стадии;
   - обработка строки прекращается;
   - pending не создаётся.

2. **Отложенная обработка (pending)**
   - создаётся/регистрируется pending-запись;
   - для pending формируется lookup-key и сериализованный payload состояния;
   - инкрементируется счётчик попыток;
   - при превышении лимита попыток pending помечается конфликтным и строка завершается ошибкой;
   - иначе формируется предупреждение о pending;
   - дальнейшее поведение зависит от partial-политики:
     - partial запрещён -> остановка без результата;
     - partial разрешён -> продолжение, результат может быть сформирован.

### 5.6 Selective validation изменённых полей по схеме целевой стороны

Стадия умеет запускать проверку схемы целевой стороны только для полей, изменённых внутри самой стадии.

Поддерживаемые возможности:

1. Проверка отключается, если схема не передана.
2. Проверка отключается, если стадия не изменила ни одного поля.
3. Проверка выполняется только по изменённым полям (после merge и/или link resolution).
4. Проверяются типы полей.
5. Нарушения схемы конвертируются в диагностику стадии и останавливают обработку строки.

Важная граница: стадия не валидирует весь результат целиком, а только свои мутации.

### 5.7 Определение целевого идентификатора записи

Стадия вычисляет целевой идентификатор по разным стратегиям в зависимости от статуса сопоставления:

1. Если предыдущая стадия уже нашла существующую запись, целевой идентификатор берётся из найденной записи.
2. Если совпадение не найдено, стадия использует:
   - целевой идентификатор, уже лежащий в результате сопоставления (если есть);
   - иначе карту заранее назначенных идентификаторов по значению идентичности.
3. Если идентификатор определить не удалось, формируется ошибка, и результат не строится.

### 5.8 Принятие решения о типе операции (создать/обновить/пропустить)

Стадия определяет тип операции по статусу сопоставления и сравнению состояний.

Поддерживаемое поведение:

1. Если совпадение не найдено -> операция создания.
2. Если совпадение найдено -> проверка возможности быстрого пропуска по fingerprint.
3. Если быстрый пропуск невозможен/не сработал -> вычисление diff.
4. Если diff пустой -> пропуск.
5. Если diff непустой -> обновление.

#### Условия быстрого пропуска по fingerprint

Быстрый пропуск применяется только при одновременном выполнении условий:

1. Есть точное/принятое совпадение с существующей записью.
2. Есть список полей fingerprint.
3. Не настроена merge-политика.
4. Не настроена кастомная diff-политика.

Если условия выполнены, сравнивается fingerprint по одному и тому же набору полей в желаемом и существующем состоянии.

### 5.9 Вычисление diff

Если быстрый пропуск не применим, стадия строит diff одним из способов:

1. Через кастомную diff-политику (если настроена).
2. Через встроенный diff по умолчанию:
   - сравнивает поля желаемого состояния с существующим;
   - игнорирует служебные поля с определённым префиксом;
   - включает только реально изменившиеся значения.

### 5.10 Формирование дополнительных данных результата (source reference и секреты)

Стадия дополняет результат планирования вспомогательной информацией для downstream-применения:

1. Source reference, сформированный из идентичности (если такая политика настроена).
2. Список secret-полей, зависящий от типа операции.
3. Политика lifecycle секретов, сериализованная в структуру результата.

### 5.11 Формирование итогового результата стадии

После успешного прохождения всех проверок стадия собирает итоговый план действия, включающий:

1. Ссылку на исходную строку.
2. Идентичность.
3. Тип операции.
4. Финальное желаемое состояние.
5. Existing-снимок (если был найден).
6. Diff изменений (для обновления).
7. Целевой идентификатор.
8. Source reference.
9. Список secret-полей.
10. Политику lifecycle секретов.

После этого, если pending не создавался и runtime-порт доступен, стадия помечает исходную строку как полностью resolved.

### 5.12 Построение индекса текущего батча для ускоренного link resolution

Стадия умеет строить вспомогательный индекс "identity-ключ -> список целевых идентификаторов" для записей текущего батча.

Назначение:

1. Разрешать ссылки между записями одного батча без обращения к внешнему runtime-хранилищу.
2. Переиспользовать те же lookup-ключи и dedup-ключи, что применяются в link resolution.

Поддерживаемые особенности:

1. Индекс строится только для правил, направленных в конкретный целевой набор данных.
2. Список индексируемых ключей включает:
   - основные lookup-ключи ссылочных правил;
   - ключи из правил сужения кандидатов.
3. Идентификатор записи для индекса может извлекаться из:
   - existing-снимка (для уже найденной записи);
   - желаемого состояния;
   - заранее назначенного целевого идентификатора.
4. На этапе чтения из индекса дубликаты кандидатов устраняются.

### 5.13 Сериализация pending payload для последующего replay

При создании pending-записи стадия сериализует расширенный снимок состояния строки в JSON.

Что включается в payload:

1. Идентичность и все её значения.
2. Ссылка на строку.
3. Текущее желаемое состояние.
4. Existing-снимок.
5. Fingerprint и список полей fingerprint.
6. Типизированное решение сопоставления (включая candidates/selected/score/meta).
7. Ссылки, извлечённые на стороне источника.
8. Целевой идентификатор (если уже есть).
9. Метаданные строки (включая переопределения ключей для ссылок).

Назначение: дать возможность позже повторно прогнать строку через стадию разрешения без потери контекста.

### 5.14 Толерантная десериализация pending payload (в пакете стадии)

В пакете стадии реализована поддержка обратной операции: чтение списка pending-записей из storage и восстановление типизированных результатов для повторной обработки.

Поддерживаемые возможности:

1. Пакетная десериализация списка pending-записей.
2. Мягкая обработка невалидных записей:
   - невалидный JSON;
   - отсутствие обязательных полей;
   - неверные типы полей;
   - неизвестный статус решения сопоставления.
3. Невалидные записи не выбрасывают исключение наружу, а считаются как `skipped`.
4. Возвращается:
   - список валидно восстановленных типизированных результатов;
   - счётчик пропущенных записей.
5. Восстанавливаются explainability-данные, source links и target id, если payload валиден.
6. Для восстановленной записи создаётся минимальный record-контейнер, достаточный для повторного прохождения pipeline.

### 5.15 Тонкая engine-обвязка стадии

Отдельная runtime-обвязка стадии выполняет:

1. Компиляцию конфигурации стадии в исполняемые правила.
2. Сборку runtime-ядра с зависимостями.
3. Поддержку переходного периода с двумя путями инициализации (через контекст исполнения и через разрозненные параметры).
4. Делегирование всех бизнес-решений ядру без собственной логики планирования.

## 6. Алгоритмы принятия решений (пошагово)

### 6.1 Главный алгоритм обработки одной строки

1. При необходимости выполнить периодический sweep просроченных pending-записей (по интервалу и при наличии runtime-порта).
2. Прочитать статус результата сопоставления.
3. Если статус конфликтный или неоднозначный:
   - сформировать ошибку стадии разрешения;
   - завершить обработку без результата.
4. Скопировать исходное желаемое состояние.
5. Если настроена merge-политика:
   - применить merge к копии состояния;
   - восстановить явно заданные значения источника при попытке перезаписи;
   - зафиксировать изменённые поля.
6. Выполнить разрешение ссылочных полей:
   - получить флаг создания pending;
   - получить флаг необходимости остановки;
   - получить список изменённых полей.
7. Если link resolution потребовал остановки — завершить обработку (с ошибками/предупреждениями).
8. Если есть изменённые полями стадии и передана схема sink — выполнить selective validation.
9. Если validation вернула нарушения — завершить обработку без результата.
10. Вычислить целевой идентификатор.
11. Если целевой идентификатор отсутствует — завершить обработку с ошибкой.
12. Определить тип операции и diff.
13. При необходимости сформировать source reference.
14. Определить набор secret-полей по типу операции.
15. Сериализовать политику lifecycle секретов в поле результата.
16. Собрать итоговый план действия.
17. Если pending не создавался и runtime-порт доступен — пометить исходную строку как resolved.
18. Вернуть результат + список ошибок/предупреждений.

### 6.2 Алгоритм link resolution с fallback batch-index -> runtime-port

1. Если ссылочные правила отсутствуют — завершить link resolution без изменений.
2. Если ссылочные правила есть, но runtime-порт не настроен — сформировать конфигурационную ошибку и остановиться.
3. Для каждого ссылочного поля:
   - проверить наличие поля в желаемом состоянии;
   - пропустить `None`;
   - при определённых условиях пропустить уже числовое значение (считать уже resolved);
   - собрать lookup-значения (meta overrides -> значения состояния);
   - по каждому lookup-ключу выполнить поиск кандидатов:
     1. сначала в индексе текущего батча;
     2. затем в runtime-порту.
4. Если по lookup найден один кандидат — записать resolved значение в поле (с coercion при необходимости).
5. Если найдено несколько кандидатов — запустить сужение по дополнительным правилам.
6. Если после сужения один кандидат — записать resolved значение.
7. Если ссылка не разрешилась — передать управление в политику pending/hard-error.
8. Продолжить цикл по следующим ссылочным полям, если текущая ситуация не требует остановки всей строки.

### 6.3 Алгоритм сужения кандидатов по доп. правилам

1. Взять исходный список кандидатов.
2. Если правил сужения нет — вернуть исходный список.
3. Для каждого правила сужения:
   - стартовать с текущего множества кандидатов;
   - для каждого ключа внутри правила определить lookup-значение;
   - получить список кандидатов по этому lookup-значению;
   - пересечь множества кандидатов.
4. Если пересечение пустое — правило не помогает (для текущего шага может остаться пусто).
5. Если пересечение непустое — обновить текущее множество кандидатов.
6. Если остался один кандидат — завершить сужение досрочно.
7. Вернуть итоговый список кандидатов.

### 6.4 Алгоритм pending lifecycle при нерешённой ссылке

1. Определить политику поля при нерешённой ссылке.
2. Если политика требует hard-error:
   - сформировать ошибку;
   - остановить обработку строки;
   - pending не создавать.
3. Иначе (pending-политика):
   - пометить, что pending создан в рамках этой строки;
   - вычислить срок истечения (или `None`, если TTL отключён);
   - сериализовать payload текущего состояния строки;
   - создать pending-запись через runtime-порт;
   - увеличить счётчик попыток для pending-записи;
   - если достигнут лимит попыток:
     - пометить pending как конфликтную;
     - сформировать ошибку;
     - остановить обработку строки.
4. Если лимит не достигнут:
   - сформировать предупреждение о pending;
   - если partial запрещён — остановить строку без результата;
   - если partial разрешён — продолжить обработку и дать стадии собрать итоговый план.

### 6.5 Алгоритм выбора операции (create/update/skip) и быстрый пропуск

1. Если результат сопоставления = "не найдено" -> выбрать создание.
2. Иначе проверить возможность быстрого пропуска по fingerprint (см. условия применимости).
3. Если fingerprint совпал -> выбрать пропуск без diff.
4. Иначе вычислить diff:
   - кастомной политикой, если она настроена;
   - встроенным сравнением по умолчанию, если нет.
5. Если diff пустой -> пропуск.
6. Если diff непустой -> обновление.

### 6.6 Алгоритм selective validation по мутированным полям

1. Если схема целевой стороны не передана — validation пропускается.
2. Если список полей, изменённых стадией, пуст — validation пропускается.
3. Проверить только изменённые поля по схеме и типам.
4. Если нарушений нет — продолжить обработку.
5. Если нарушения есть:
   - преобразовать нарушения в диагностику стадии;
   - остановить обработку строки.

### 6.7 Алгоритм построения batch-индекса

1. Определить набор ключей, полезных для ссылочного resolve в рамках конкретного целевого набора данных:
   - lookup-ключи;
   - ключи из правил сужения.
2. Определить, какое поле считается целевым идентификатором для этого индекса.
3. Для каждой строки батча:
   - попытаться извлечь её целевой идентификатор;
   - если идентификатора нет — пропустить;
   - для каждого индексируемого ключа извлечь значение;
   - сформировать identity-ключ и добавить в индекс.
4. Вернуть структуру индексирования по dataset -> identity_key -> список ids.

### 6.8 Алгоритм толерантной загрузки pending-записей из storage

1. Для каждой pending-записи попытаться распарсить JSON payload.
2. При ошибке JSON — увеличить счётчик skipped и перейти к следующей записи.
3. Проверить наличие и типы обязательных полей (идентичность, row-ref, состояние, fingerprint, решение сопоставления и т.д.).
4. Проверить, что статус решения сопоставления известен и распознаётся.
5. Восстановить типизированное решение, кандидатов и source-links.
6. Восстановить типизированный результат сопоставления.
7. Сформировать минимальный контейнер результата конвейера для replay.
8. Добавить в список валидных результатов.
9. Вернуть список результатов + количество пропущенных записей.

## 7. Ошибки, предупреждения и условия остановки/пропуска

### Ошибки, формируемые стадией

1. **Конфликтный/неоднозначный результат сопоставления**
   - Когда возникает: предыдущая стадия не смогла однозначно определить целевую запись.
   - Эффект: план действия не строится.

2. **Конфигурационная ошибка runtime для ссылочного resolve**
   - Когда возникает: нужны ссылочные lookup/pending операции, но runtime-порт не настроен.
   - Эффект: строка завершается ошибкой.

3. **Нерешённая ссылка с политикой hard-error**
   - Когда возникает: не найден кандидат или кандидаты неоднозначны, а поле требует hard-error.
   - Эффект: строка завершается ошибкой без pending.

4. **Достигнут лимит попыток pending**
   - Когда возникает: после создания/обновления pending счётчик попыток достигает или превышает лимит.
   - Эффект: pending помечается конфликтным, строка завершается ошибкой.

5. **Невозможно определить целевой идентификатор**
   - Когда возникает: статус/данные требуют target id, но он не найден ни в existing, ни во входных map-подсказках.
   - Эффект: результат не строится.

6. **Нарушение схемы sink на изменённых полях**
   - Когда возникает: после merge/link-mutation изменённые поля не проходят проверку типа/схемы.
   - Эффект: результат не строится.

### Предупреждения, формируемые стадией

1. **Ссылка отложена (pending)**
   - Когда возникает: ссылка не разрешилась и для поля настроен pending-режим, а лимит попыток ещё не достигнут.
   - Эффект: строка либо откладывается, либо продолжается в partial-режиме.

### Условия остановки без результата (не всегда ошибка)

1. Pending создан, partial-режим запрещён -> строка останавливается без результата, но с предупреждением.
2. Конфликт/неоднозначность после стадии сопоставления -> строка останавливается с ошибкой.

### Условия успешного продолжения при частичной готовности

1. Нерешённая ссылка допустима в partial-режиме -> результат строится, но поле может остаться в исходном значении.

### Поведение pending replay-декодера при ошибках данных

1. Ошибки payload не выбрасываются как исключения наружу.
2. Невалидные записи считаются `skipped`.
3. Решение о логировании/наблюдаемости skipped принимает вызывающая сторона, а не декодер.

## 8. Побочные эффекты и технические взаимодействия

### Чтение из внешних компонентов

1. Поиск кандидатов по identity-ключам через runtime-порт.
2. Sweep просроченных pending-записей.
3. Проверка схемы целевой стороны через валидатор схемы (если передана схема).

### Запись во внешние компоненты

1. Создание pending-записей.
2. Увеличение счётчика попыток pending.
3. Пометка pending как конфликтной записи.
4. Пометка исходной строки как resolved (без pending).

### Локальные побочные эффекты

1. Обновление времени последнего sweep.
2. Накопление буфера просроченных pending-записей.
3. Логирование предупреждения при попытке merge-политики перезаписать явные значения источника.
4. Мутация локальной копии желаемого состояния (входное состояние не меняется).

### Работа с сериализацией

1. Сериализация pending payload в JSON при создании pending.
2. Десериализация pending payload в отдельном модуле пакета стадии (для replay).

### Что стадия не делает

1. Не применяет изменения в target.
2. Не обновляет identity-индекс после успешного применения (это задача downstream-этапа после apply).
3. Не принимает решения о долгосрочном хранении/очистке pending beyond sweep trigger (часть параметров настроек используется вне ядра стадии).

## 9. Влияние конфигурации (DSL/spec/compiler options/runtime settings)

### Конфигурация сборки желаемого состояния и diff/merge

Влияет на:

1. Какие поля попадают в состояние сравнения/применения.
2. Какие поля и как сравниваются при diff.
3. Какие пустые значения могут дозаполняться из existing.
4. Нормализацию значений при diff/merge (текст/булево).

Следствие:

1. Меняет чувствительность стадии к различиям данных.
2. Меняет число операций update vs skip.
3. Меняет список полей, которые могут быть изменены ещё до link resolution.

### Конфигурация ссылочного resolve

Влияет на:

1. Какие поля считаются ссылочными.
2. В какой целевой набор данных выполняется lookup.
3. По каким lookup-ключам искать и в каком порядке.
4. Какие доп. правила сужения применяются при множественных кандидатах.
5. Требуется ли приведение типа resolved id.
6. Что делать при нерешённой ссылке (pending/hard-error).

### Конфигурация секретов и lifecycle

Влияет на:

1. Какие поля считать секретными при разных типах операций.
2. Какую lifecycle-политику передать downstream (persistent/ephemeral, delete-on-success, TTL).
3. Значения по умолчанию lifecycle при отсутствии явных настроек.

### Конфигурация source-reference

Влияет на:

1. Нужно ли формировать source reference.
2. Какие поля идентичности включать в source reference.
3. Нужно ли принудительно включать primary-ключ даже при неполном списке полей.

### Конфигурационные ограничения компилятора

Влияют на допустимость конфигурации ещё до runtime:

1. Возможность запретить pending links на уровне compile-options.
2. Проверка обязательности `desired_state` и `diff` блоков.
3. Проверка корректности link-правил (непустые lookup-ключи, непустые dedup-правила, отсутствие дубликатов полей ссылок).
4. Требование схемы sink при включённой генерации diff-правил из схемы.

### Runtime-настройки pending-механики

Влияют на:

1. TTL для pending-записей.
2. Лимит попыток обработки pending.
3. Интервал запуска sweep просроченных pending.
4. Разрешён ли partial-режим при pending.
5. Поля настроек, используемые вне ядра стадии (например, политика on-expire и retention) — влияют на окружающие процессы/maintenance.

### Переходная engine-обвязка

Поддерживаются два пути инициализации ядра стадии:

1. Через общий контекст исполнения (новый путь).
2. Через разрозненные зависимости (legacy/transition).

Это влияет на сборку объекта стадии, но не должно менять семантику алгоритмов.

## 10. Что относится к бизнес-логике данных

### Ядро логики данных (основная ценность стадии)

1. Гейт по статусу сопоставления (когда разрешение невозможно по смыслу).
2. Merge/fill-from-existing правила с защитой приоритетов источника.
3. Разрешение ссылок по lookup-ключам и правилам сужения кандидатов.
4. Политика обработки нерешённых ссылок (pending vs hard-error, partial vs stop).
5. Вычисление типа операции (создать/обновить/пропустить).
6. Семантика быстрого пропуска по fingerprint.
7. Вычисление diff изменений.
8. Формирование source reference и секретной политики в результате планирования.

### Техническая поддержка стадии (полезно, но не ядро логики данных)

1. Периодический sweep просроченных pending.
2. Буферизация результатов sweep и метод их выдачи.
3. Selective validation изменённых полей по схеме sink.
4. Сериализация pending payload для replay.
5. Десериализация pending payload (в отдельном модуле пакета).
6. Формирование диагностических объектов и интеграция с каталогом.
7. Тонкая engine-обвязка и компиляция конфигурации.

### Спорные зоны (межслойное смешение)

1. Сериализация storage-payload внутри ядра бизнес-решений.
2. Housekeeping pending (sweep) в горячем пути обработки строки.
3. Проверка схемы sink внутри ядра стадии планирования.
4. Transition-логика двойной инициализации в engine-обвязке.

## 11. Спорные и лишние ответственности (классификация + рекомендации)

### 11.1 Логирование предупреждения о попытке merge-политики перезаписать явные значения источника

- Классификация: `Diagnostics presentation concern`
- Наблюдение: ядро стадии не только защищает приоритет данных источника, но и напрямую пишет предупреждение в лог.
- Текущая ценность для стадии: событие фиксируется рядом с местом нарушения контракта merge-политики.
- Почему это спорно/лишнее с точки зрения границ: доменная логика смешивается с механизмом наблюдаемости (логгер, формат сообщения).
- Риск при оставлении как есть: усложняется переиспользование ядра и замена наблюдаемости; лог-формат становится частью неявного контракта ядра.
- Рекомендуемое место ответственности: mapper доменных событий стадии -> logging/telemetry в engine/use-case слое.
- Минимальный шаг выделения: возвращать/эмитить типизированное domain-warning событие вместо прямого логирования.
- Приоритет: `Medium`
- Влияние на поведение: бизнес-результат не изменится.

### 11.2 Сериализация pending payload в JSON внутри ядра стадии

- Классификация: `Serialization/DTO concern`
- Наблюдение: ядро стадии формирует storage-представление pending-записи (JSON payload) в деталях, включая сериализацию вложенных структур решения сопоставления.
- Текущая ценность для стадии: полный контекст pending создаётся сразу в момент решения, без дополнительного слоя.
- Почему это спорно/лишнее с точки зрения границ: сериализация storage-формата — это отдельная ответственность представления/DTO, не ядро алгоритма разрешения данных.
- Риск при оставлении как есть: изменения формата pending payload затрагивают ядро бизнес-логики; сложнее развивать формат версии payload независимо.
- Рекомендуемое место ответственности: отдельный codec/serializer компонент pending payload.
- Минимальный шаг выделения: вынести сериализацию payload и вложенных структур в отдельный модуль/сервис, оставив ядру только подготовку доменного снимка.
- Приоритет: `High`
- Влияние на поведение: не должно измениться при сохранении формата payload.

### 11.3 Проверка схемы sink по изменённым полям внутри ядра стадии

- Классификация: `Cross-layer leakage`
- Наблюдение: ядро стадии после собственных мутаций вызывает валидатор схемы целевой стороны и маппит его ошибки в диагностику.
- Текущая ценность для стадии: защищает от некорректных мутаций ссылок/merge до стадии применения.
- Почему это спорно/лишнее с точки зрения границ: схема sink и её валидация находятся на границе домена и контракта внешней системы; ядро стадии планирования начинает знать слишком много о схеме sink и диагностике DSL.
- Риск при оставлении как есть: рост связности между стадией разрешения, схемой sink и DSL-диагностикой; усложнение независимого тестирования.
- Рекомендуемое место ответственности: boundary validation слой вокруг ядра стадии (planner adapter/use-case orchestration).
- Минимальный шаг выделения: ядро возвращает список мутированных полей и результат без схемной валидации; внешний слой вызывает validation и при необходимости отклоняет результат.
- Приоритет: `High`
- Влияние на поведение: не должно измениться при корректном переносе.

### 11.4 Housekeeping sweep просроченных pending внутри горячего пути обработки строки

- Классификация: `Stage technical support`
- Наблюдение: перед обработкой строки ядро может запускать sweep просроченных pending по таймеру.
- Текущая ценность для стадии: не требует отдельного maintenance-триггера, обеспечивает постепенную очистку в боевом потоке.
- Почему это спорно/лишнее с точки зрения границ: операция housekeeping не связана с решением по конкретной строке и добавляет операционные побочные эффекты в основной алгоритм.
- Риск при оставлении как есть: непредсказуемое время обработки отдельных строк, смешение бизнес-логики и обслуживания состояния.
- Рекомендуемое место ответственности: отдельный maintenance hook/use-case/планировщик, вызываемый вне обработки строки.
- Минимальный шаг выделения: вынести sweep в отдельный публичный метод orchestration-уровня и перестать вызывать его автоматически из обработки строки.
- Приоритет: `Medium`
- Влияние на поведение: поведение по строкам не изменится; изменится момент sweep.

### 11.5 Тесная связка ядра с каталогом диагностик и форматированием сообщений

- Классификация: `Diagnostics presentation concern`
- Наблюдение: ядро стадии непосредственно создаёт ошибки/предупреждения в формате общей подсистемы диагностики.
- Текущая ценность для стадии: точная диагностика формируется рядом с веткой, где принимается решение.
- Почему это спорно/лишнее с точки зрения границ: ядро зависит от конкретной диагностической модели, кодов и helper-функций, вместо возврата типизированных причин доменного отказа.
- Риск при оставлении как есть: усложнение миграции диагностического слоя и переиспользования алгоритма вне текущей диагностической инфраструктуры.
- Рекомендуемое место ответственности: mapper domain-outcome -> diagnostics на уровне orchestration.
- Минимальный шаг выделения: постепенно заменять прямую генерацию диагностик на типизированные причины, оставляя адаптер совместимости.
- Приоритет: `Medium`
- Влияние на поведение: внешнее поведение не изменится при сохранении тех же кодов и сообщений.

### 11.6 Сериализация lifecycle политики секретов в структуру результата внутри ядра стадии

- Классификация: `Serialization/DTO concern`
- Наблюдение: ядро стадии конвертирует внутреннюю lifecycle-политику секретов в сериализуемый словарь, пригодный для downstream.
- Текущая ценность для стадии: downstream получает простой DTO без знания внутренних типов политики.
- Почему это спорно/лишнее с точки зрения границ: это преобразование доменная-модель -> transport DTO, а не правило данных разрешения.
- Риск при оставлении как есть: изменения формата DTO затрагивают ядро стадии; усложняется эволюция внутренней модели политики.
- Рекомендуемое место ответственности: mapper результата стадии в boundary/adapter слое либо общий serializer планового результата.
- Минимальный шаг выделения: возвращать типизированную lifecycle-модель в результате ядра и сериализовать на границе.
- Приоритет: `Low`
- Влияние на поведение: не должно измениться.

### 11.7 Переходная dual-init логика в engine-обвязке (контекст vs разрозненные параметры)

- Классификация: `Transitional compatibility`
- Наблюдение: engine-обвязка поддерживает одновременно новый и legacy пути сборки зависимостей.
- Текущая ценность для стадии: упрощает постепенную миграцию существующих вызовов.
- Почему это спорно/лишнее с точки зрения границ: это не функциональная возможность стадии разрешения данных, а временная совместимость инициализации.
- Риск при оставлении как есть: конструктор engine становится перегруженным, повышается вероятность конфигурационных расхождений между путями.
- Рекомендуемое место ответственности: отдельный compatibility factory/adapter вне канонического engine-конструктора.
- Минимальный шаг выделения: оставить один канонический путь внутри engine, legacy-путь вынести в фабрику совместимости.
- Приоритет: `Low`
- Влияние на поведение: не должно измениться.

### 11.8 Толерантный pending replay-декодер внутри пакета стадии разрешения

- Классификация: `Stage technical support`
- Наблюдение: в пакете стадии размещён декодер storage-payload для replay pending, который восстанавливает типизированный результат сопоставления.
- Текущая ценность для стадии: логика replay рядом с форматом payload, проще поддерживать симметрию чтения/записи.
- Почему это спорно/лишнее с точки зрения границ: это функция взаимодействия со storage-представлением и replay-потоком, а не ядро решения по данным строки.
- Риск при оставлении как есть: пакет стадии становится контейнером для смешанных runtime/storage concerns.
- Рекомендуемое место ответственности: отдельный модуль replay/pending-codec в слое planning-runtime или boundary adapters.
- Минимальный шаг выделения: переместить декодер в нейтральный пакет и оставить re-export в пакете стадии.
- Приоритет: `Low`
- Влияние на поведение: не должно измениться.

## 12. Рекомендации по декомпозиции/переносу ответственности

### Приоритетные шаги (без изменения поведения)

1. **Вынести pending payload serialization из ядра стадии**
   - Оставить ядру подготовку доменного снимка pending.
   - Отдельный codec сериализует/версирует payload.

2. **Вынести sink schema validation из горячего пути ядра**
   - Ядро возвращает результат + список полей, изменённых стадией.
   - Оркестратор/adapter выполняет схемную проверку и добавляет диагностику.

3. **Отделить pipeline housekeeping (sweep) от обработки строки**
   - Сделать sweep отдельным вызываемым шагом orchestration уровня.
   - Оставить ядру только доступный API чтения накопленных событий/expired записей при необходимости.

### Среднесрочные шаги

1. Вынести логирование merge-конфликта policy в mapper domain-events -> telemetry.
2. Вынести сериализацию lifecycle секретов в boundary mapper результата стадии.
3. Переместить pending replay codec в нейтральный planning/runtime пакет.

### Переходные меры совместимости

1. Сохранить формат pending payload неизменным на период вынесения codec.
2. Сохранить текущие диагностические коды и тексты при вынесении mapper диагностик.
3. Сохранить re-export публичных импортов на период перемещения модулей.

## 14. Проверочные сценарии (чеклист покрытия)

Ниже перечислены сценарии, которые должны быть отражены в анализе стадии. Для каждого сценария указано, чем он подтверждён: тестами, кодом или обоими.

5. Unresolved link -> pending warning + stop — подтверждено unit-тестом.
6. Unresolved link -> pending warning + partial continue — подтверждено unit-тестом.
7. Unresolved link -> hard error (без pending) — подтверждено unit-тестом.
8. Достижение max attempts -> конфликт и остановка — подтверждено unit-тестом.
9. Sink type validation на мутированном link поле — подтверждено unit-тестом.
10. Решение `create` — подтверждено кодом (явная ветка выбора операции); отдельного теста в указанном наборе не найдено.

12. Решение `update` через diff policy/default diff — подтверждено кодом (явные ветки); отдельного теста в указанном наборе не найдено.
