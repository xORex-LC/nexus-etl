# Matcher: Полная карта функциональных возможностей и аудит ответственности

## Кратко

Этот документ описывает фактическое поведение стадии сопоставления записей на основе кода и тестов: какие решения она принимает, какие ветки алгоритмов поддерживает, какие побочные эффекты создаёт, и какие части её поведения относятся к данным, а какие являются технической/переходной нагрузкой.

Основной текст намеренно описывает поведение в терминах бизнес-смысла и алгоритмов без привязки к именам сущностей в коде. Точные ссылки на файлы, модули и функции собраны в отдельном разделе трассировки.

## 1. Назначение стадии и границы ответственности

### Что стадия делает

Стадия сопоставления получает строку после предыдущих стадий подготовки данных и решает, существует ли соответствующая запись в целевой стороне.

Она выполняет четыре ключевых задачи:

1. Определяет идентичность записи по набору правил и приоритетов.
2. Ищет совпадение по точному ключу.
3. При необходимости применяет нестрогое (score-based) сопоставление по ограниченному набору кандидатов.
4. Формирует объяснимый результат сопоставления для следующей стадии (включая причину решения, кандидатов, score и подготовленное желаемое состояние).

### Что стадия не должна делать (архитектурная граница)

1. Не должна выполнять операции создания/изменения данных в целевой системе.
2. Не должна принимать решение о типе итоговой операции синхронизации (создать/обновить/пропустить).
3. Не должна управлять жизненным циклом отложенных ссылок.
4. Не должна реализовывать инфраструктуру хранения/поиска (она использует порт).

### Фактическое отклонение от идеальной границы (есть в реализации)

Стадия частично готовит данные для следующей стадии (собирает желаемое состояние) и частично управляет runtime-состоянием дедупликации источника. Эти зоны отдельно разобраны в аудите ответственности.

## 2. Что стадия получает на вход

### На уровне одной записи

Стадия получает контейнер результата предыдущего шага конвейера, в котором может быть:

1. Исходная запись после нормализации/обогащения.
2. Ссылка на исходную строку (может быть неполной или отсутствовать).
3. Составной ключ для сопоставления (может отсутствовать).
4. Диагностика, накопленная предыдущими стадиями.
5. Метаданные (включая сведения о секретных полях).
6. Набор кандидатов секретных значений, переданный сверху.

### На уровне конфигурации/окружения

Стадия использует:

1. Название набора данных (для поиска в кэше и namespace runtime-state).
2. Порт runtime-доступа к кэшу/индексу и runtime-state.
3. Правила сопоставления (identity, дедупликация в источнике, fuzzy/scoring, игнорируемые поля).
4. Правила сборки желаемого состояния (поступают из конфигурации следующей стадии, но вызываются здесь).
5. Флаг, разрешающий/запрещающий учитывать удалённые записи при поиске кандидатов.
6. Каталог диагностик для формирования ошибок/предупреждений.

### Важные предпосылки входа

1. Входная запись может отсутствовать (например, если предыдущая стадия уже отбросила строку).
2. Поля, используемые для identity/fuzzy, могут быть частично пустыми.
3. Кэш может вернуть 0, 1 или несколько кандидатов.
4. Runtime-state дедупликации может быть как локальным (в памяти экземпляра), так и разделяемым через scoped state.

## 3. Что стадия возвращает на выход

### Основной результат

Стадия возвращает контейнер результата конвейера с сохранением исходной обвязки (record/meta/накопленные diagnostics) и добавляет в него результат сопоставления одной строки.

### Возможные варианты результата

1. **Успешно обработанная строка с результатом сопоставления**
   - содержит идентичность;
   - содержит желаемое состояние для следующей стадии;
   - содержит снимок найденной целевой записи (если найдено совпадение);
   - содержит fingerprint и список полей fingerprint;
   - содержит объяснимое решение сопоставления (статус, причина, score, выбранный кандидат, top-кандидаты);
   - содержит ссылочные идентичности источника (если они были сформированы отдельным правилом).

2. **Отброшенная строка (hard drop внутри стадии)**
   - поле результата строки очищено;
   - в метаданные записана причина отбрасывания;
   - добавлена ошибка или предупреждение (в зависимости от политики);
   - остальные части контейнера сохраняются.

3. **Строка без результата из-за входной ошибки/пустой строки**
   - строка отсутствует;
   - диагностика пополняется ошибкой отсутствия идентичности/пустого входа.

## 4. Состояние, которое стадия хранит/использует во время прогона

### Локальное состояние экземпляра

1. Внутрипроцессная таблица "уже встреченных" записей источника (ключ -> fingerprint) для дедупликации внутри батча.
2. Опциональный идентификатор runtime-scope для разделяемого runtime-state.
3. Ссылки на правила сопоставления, правила сборки желаемого состояния, порт runtime-кэша, флаг включения удалённых, каталог диагностик.

### Состояние, читаемое/пишущееся через порт

1. Поиск кандидатов по фильтру в кэше/индексе.
2. Чтение fingerprint ранее обработанной записи источника в scoped runtime-state.
3. Запись fingerprint в scoped runtime-state.

### Особенности жизненного цикла состояния

1. Локальная таблица дедупликации сбрасывается перед потоковой обработкой.
2. При обработке отдельных вызовов (по одному элементу) локальная таблица сохраняется между вызовами, пока не выполнен явный сброс.
3. Scoped runtime-state позволяет разделять дедупликацию между разными экземплярами стадии в рамках одного логического прогона.

## 5. Полная карта функциональных возможностей

### 5.1 Подготовка рабочего контекста строки

Стадия умеет собрать единый рабочий контекст, даже если предыдущая стадия не передала полный набор служебных данных.

Поддерживаемое поведение:

1. Восстановление ссылки на строку по record-данным, если ссылка отсутствует.
2. Подстановка технического идентификатора первичного ключа для строки, если в ссылке не заполнено поле идентичности.
3. Перенос диагностик и метаданных из upstream в контекст выполнения.
4. Извлечение списка секретных полей:
   - сначала из метаданных;
   - если список отсутствует, то из набора кандидатов секретов.

Зачем это нужно:

1. Делает стадию устойчивой к разным входным путям (с валидацией/без валидации).
2. Обеспечивает корректную диагностику и трассировку даже на неполном входе.

### 5.2 Канонизация ссылки на строку

После выбора идентичности стадия умеет привести ссылку на строку к актуальному виду:

1. Если ссылки не было, создаёт новую с line/row id и выбранной идентичностью.
2. Если ссылка есть, но в ней устаревшие поля идентичности, создаёт корректированную копию.
3. Если ссылка уже совпадает с выбранной идентичностью, оставляет как есть.

Это важно для downstream-диагностики и единообразия ссылок на строку.

### 5.3 Определение идентичности по цепочке правил

Стадия поддерживает приоритетный перебор нескольких правил идентификации.

Поддерживаемые возможности:

1. Для каждого правила строится кандидат идентичности из данных строки и рабочего контекста.
2. Если значение первичного компонента идентичности пустое, правило пропускается.
3. Первая валидная идентичность запоминается как базовая (даже если точного совпадения пока нет).
4. Для каждого валидного кандидата выполняется точный поиск в runtime-кэше.
5. Если найден ровно один кандидат, сопоставление завершает exact-ветку успешно.
6. Если найдено несколько кандидатов, формируется конфликтная ошибка и строка прекращает дальнейшую обработку.
7. Если ни по одному правилу нет валидной идентичности, формируется ошибка отсутствия идентичности.
8. Если идентичность есть, но совпадение не найдено, стадия возвращает результат "не найдено" и может перейти к fuzzy-ветке (если она включена).

### 5.4 Точный поиск совпадений в кэше

Стадия выполняет точный lookup по паре "поле идентичности -> значение".

Особенности:

1. Поиск выполняется через порт, а не напрямую через инфраструктуру.
2. Поведение зависит от флага включения удалённых записей.
3. Множественный результат трактуется как конфликт качества данных/индекса.
4. Единственный результат считается достаточным основанием для exact-match.

### 5.5 Формирование желаемого состояния для следующей стадии

Стадия формирует "желаемое состояние" записи (набор полей, который нужно сравнивать/применять дальше).

Фактически поддерживается:

1. Вызов отдельного правила сборки состояния на основе строки и рабочего контекста.
2. Независимость от exact/fuzzy результата (состояние формируется до финальной сборки результата).
3. Использование этого состояния как источника данных для fingerprint и fuzzy scoring.

Это функционально полезно, но архитектурно спорно, потому что связывает стадию сопоставления со следующей стадией.

### 5.6 Построение fingerprint для дедупликации и быстрых downstream-решений

Стадия вычисляет хэш состояния и список полей, участвующих в этом хэше.

Поддерживаемые возможности:

1. Исключение отдельных полей из fingerprint по конфигурации.
2. Канонизация порядка полей в fingerprint (детерминизм).
3. Использование fingerprint для:
   - source-dedup внутри батча;
   - downstream-решений о пропуске изменений.

### 5.7 Формирование объяснимого результата сопоставления

Стадия возвращает не только статус, но и контекст решения:

1. Статус решения (найдено / не найдено / неоднозначно / конфликт в источнике).
2. Код причины решения (например: точное совпадение, fuzzy tie, fuzzy review, отсутствие кандидатов и т.д.).
3. Общий score (если применялась scoring-ветка).
4. Выбранный кандидат (когда решение однозначное и принято).
5. Список top-кандидатов для explainability.
6. Метаданные решения (например, режим сопоставления: exact/fuzzy).

### 5.8 Нестрогое сопоставление (fuzzy/scoring)

Если точное совпадение не найдено и scoring-режим включён, стадия применяет отдельный алгоритм ранжирования.

#### Возможности ветки кандидатов

1. Поиск кандидатов не по всем данным, а по ограничивающим ключам (blocking).
2. Перебор нескольких blocking-ключей по порядку.
3. Дедупликация кандидатов, пришедших из разных blocking-ключей.
4. Ограничение общего количества кандидатов (`max_candidates`) для контроля стоимости вычислений.
5. Мягкая обработка неизвестных blocking-полей (переходный режим: ключ пропускается, обработка продолжается).

#### Возможности ветки scoring

1. Сбор source-значений для полей scoring из нескольких источников:
   - из желаемого состояния;
   - из выбранной идентичности;
   - из самой строки.
2. Поддержка нескольких режимов сравнения полей:
   - точное;
   - без учёта регистра/нормализации;
   - similarity (строковая похожесть).
3. Весовая агрегация score по полям.
4. Игнорирование полей с нулевым весом.
5. Округление итогового score по конфигурации.
6. Консервативный fallback неизвестного режима сравнения к точному сравнению.

#### Возможности принятия решения по score

1. Детекция tie (слишком близкие top1 и top2).
2. Разделение порогов:
   - автоматическое принятие;
   - зона ручной проверки (неоднозначность);
   - отказ.
3. Возврат top-k кандидатов для explainability независимо от итогового статуса.

### 5.9 Дедупликация записей внутри источника (source dedup)

После получения результата сопоставления стадия может дополнительно отбрасывать повторы/конфликты внутри входного батча.

Поддерживаемые возможности:

1. Включаемый/выключаемый режим source-dedup.
2. Канонический ключ дедупликации, учитывающий:
   - набор данных;
   - имя первичного поля идентичности;
   - значение первичного поля идентичности.
3. Если канонический ключ построить нельзя (пустые значения), дедупликация не применяется.
4. Сохранение fingerprint первой встреченной записи по ключу.
5. Различение двух случаев:
   - повтор без изменений (одинаковый fingerprint) -> duplicate;
   - конфликтующее повторение (другой fingerprint) -> conflict.
6. Фактическое runtime-поведение для duplicate (одинаковый fingerprint):
   - запись всегда hard-drop;
   - формируется предупреждение.
7. Конфигурируемая реакция для conflict (разный fingerprint):
   - предупреждение;
   - ошибка.
8. Hard drop записи внутри стадии с сохранением причины drop в метаданных.
9. Возможность разделяемой дедупликации между экземплярами через scoped runtime-state.

### 5.10 Потоковая обработка

Стадия предоставляет потоковый режим обработки набора записей:

1. Перед началом потока сбрасывает локальное состояние дедупликации.
2. Обрабатывает записи по одной.
3. Применяет source-dedup внутри того же потока.
4. Возвращает результаты лениво (generator-style поведение).

### 5.11 Диагностическое поведение

Стадия умеет как переносить upstream-диагностику, так и добавлять собственную.

Поддерживаемые типы проблем:

1. Пустая/отсутствующая входная строка.
2. Отсутствие идентичности.
3. Конфликт нескольких целевых кандидатов по точному ключу.
4. Повтор строки в источнике.
5. Конфликтующие строки в источнике.

### 5.12 Поддерживающие модели и контракты внутри пакета стадии

Внутри пакета стадии реализованы также вспомогательные элементы, влияющие на функциональность/контракты:

1. Типизированное решение сопоставления и статусы.
2. Модели результата сопоставления и результата следующей стадии (совместное размещение).
3. Функции построения fingerprint.
4. Унифицированный формат текстового identity-ключа (используется и другой стадией).
5. Тонкая engine-обвязка: компиляция конфигурации и сборка runtime-ядра.

## 6. Алгоритмы принятия решений (пошагово)

### 6.1 Главный алгоритм обработки одной строки

1. Извлечь строку и собрать рабочий контекст из входного контейнера.
2. Если строки нет:
   - вернуть контейнер без результата строки;
   - если upstream не передал ошибок, добавить ошибку "пустая/отсутствующая строка".
3. Определить идентичность и попытаться найти точное совпадение по цепочке правил.
4. Если на этапе определения идентичности возникла конфликтная ошибка (несколько кандидатов по точному ключу), вернуть строку как ошибочную без результата стадии.
5. Если идентичность отсутствует или её первичное значение пусто, вернуть ошибку отсутствия идентичности.
6. Сформировать желаемое состояние для downstream.
7. Построить fingerprint и список полей fingerprint.
8. Инициализировать explainability-поля решения (режим, score, список кандидатов, причина).
9. Если exact-совпадение найдено:
   - отметить статус как "найдено";
   - выставить score=1.0;
   - указать причину "точное совпадение";
   - сформировать выбранного кандидата и top-кандидатов.
10. Иначе, если включён fuzzy/scoring:
   - собрать ограниченный набор кандидатов;
   - ранжировать по score;
   - определить статус/причину по tie и порогам;
   - подготовить top-кандидатов.
11. Иначе (fuzzy выключен): вернуть статус "не найдено" с причиной "точное совпадение не найдено".
12. Сформировать explainable-решение.
13. Канонизировать ссылку на строку под выбранную идентичность.
14. Сформировать результат сопоставления (идентичность, existing, desired_state, fingerprint, объяснение, ссылки).
15. Вернуть исходный контейнер с добавленным результатом стадии.

### 6.2 Алгоритм выбора идентичности по приоритетным правилам

1. Взять список правил идентификации в заданном порядке.
2. Для каждого правила собрать кандидата идентичности.
3. Если первичное значение кандидата пустое, перейти к следующему правилу.
4. Если это первый валидный кандидат, временно сохранить его как базовую идентичность строки.
5. Выполнить точный поиск кандидатов в кэше.
6. Если найдено больше одного кандидата:
   - вернуть конфликтную ошибку;
   - остановить выбор.
7. Если найден ровно один кандидат:
   - вернуть эту идентичность и найденную запись;
   - статус exact-match.
8. Если совпадения нет, продолжить цикл (возможен матч по следующему правилу).
9. После завершения цикла:
   - если валидной идентичности не было вовсе -> ошибка отсутствия идентичности;
   - если валидная идентичность была, но совпадение не найдено -> статус "не найдено".

### 6.3 Алгоритм fuzzy/scoring decision matrix

1. Проверить, задан ли список blocking-ключей. Если нет, кандидатов нет.
2. По каждому blocking-ключу попытаться извлечь значение из:
   - желаемого состояния;
   - выбранной идентичности;
   - исходной строки.
3. Если значение пустое, этот blocking-ключ пропускается.
4. Запросить кандидатов из runtime-кэша по blocking-ключу.
5. Объединить кандидатов из разных blocking-ключей с дедупликацией.
6. Ограничить общее число кандидатов конфигурационным лимитом.
7. Если кандидатов нет -> статус "не найдено", причина "нет кандидатов для fuzzy".
8. Собрать source-values для полей scoring.
9. Для каждого кандидата вычислить weighted score.
10. Отсортировать кандидатов по score по убыванию.
11. Если ранжированный список пуст -> статус "не найдено", причина "нет ранжированных кандидатов".
12. Сформировать top-k кандидатов для explainability.
13. Проверить tie между top1 и top2 по допустимой дельте:
   - если tie -> статус "неоднозначно", причина "tie".
14. Сравнить лучший score с порогом автоматического принятия:
   - если выше/равен -> статус "найдено".
15. Иначе сравнить с порогом review:
   - если выше/равен -> статус "неоднозначно" (требует ручной проверки).
16. Иначе -> статус "не найдено" (score недостаточен).

### 6.4 Алгоритм дедупликации источника и правила drop

1. Выполнить обычное сопоставление строки.
2. Если результат строки отсутствует (ошибка/пустой вход), source-dedup не применяется.
3. Если source-dedup выключен конфигурацией, вернуть результат без изменений.
4. Построить канонический ключ дедупликации по набору данных + выбранной идентичности.
5. Если канонический ключ не получился, вернуть результат без изменений.
6. Прочитать ранее сохранённый fingerprint:
   - сначала из scoped runtime-state (если scope задан);
   - затем из локальной памяти экземпляра.
7. Если это первое появление ключа:
   - сохранить текущий fingerprint локально и (при наличии scope) в runtime-state;
   - вернуть результат без изменений.
8. Если fingerprint совпал с ранее сохранённым:
   - сформировать предупреждение о дубле;
   - отбросить строку (очистить payload результата строки);
   - записать причину drop в метаданные.
9. Если fingerprint отличается:
   - при политике "warn" сформировать предупреждение и отбросить строку;
   - при политике "error" сформировать ошибку и отбросить строку.

### 6.5 Алгоритм сборки explainable результата

1. Определить тип решения (найдено/не найдено/неоднозначно/конфликт источника).
2. Выбрать reason-код по ветке exact/fuzzy.
3. Сформировать выбранного кандидата, если решение однозначное и имеется конкретная целевая запись.
4. Сформировать список top-кандидатов (даже при неоднозначности/отказе fuzzy).
5. Сохранить score (если он вычислялся).
6. Сохранить метаданные режима сопоставления (`exact`/`fuzzy`).
7. Упаковать все поля в типизированную структуру решения для downstream.

## 7. Ошибки, предупреждения и условия остановки/пропуска

### Ошибки, формируемые стадией

1. **Пустая/отсутствующая строка на входе**
   - Когда возникает: upstream передал контейнер без строки и без ошибок.
   - Эффект: результат строки отсутствует, обработка этой строки прекращается.

2. **Идентичность не определена**
   - Когда возникает: ни одно правило не дало валидного первичного значения.
   - Эффект: строка не получает результат сопоставления.

3. **Конфликт точного поиска по идентичности**
   - Когда возникает: по точному ключу найдено больше одной целевой записи.
   - Эффект: строка не получает результат сопоставления.

4. **Конфликт источника (режим source-dedup = error)**
   - Когда возникает: повторяющаяся запись источника имеет другой fingerprint, а политика требует ошибки.
   - Эффект: строка hard-drop внутри стадии.

### Предупреждения, формируемые стадией

1. **Дубль источника**
   - Когда возникает: повторяющаяся запись имеет тот же fingerprint.
   - Эффект: строка hard-drop с предупреждением.

2. **Конфликт источника (режим source-dedup = warn)**
   - Когда возникает: повторяющаяся запись имеет другой fingerprint, но политика допускает warning.
   - Эффект: строка hard-drop с предупреждением.

### Состояния без ошибки, но с "не найдено"

Это нормальный результат, а не ошибка стадии:

1. Точное совпадение отсутствует, fuzzy выключен.
2. Точное совпадение отсутствует, fuzzy включён, но нет кандидатов.
3. Кандидаты есть, но лучший score ниже порога review.

### Состояния "неоднозначно"

Это не обязательно ошибка на этом этапе (строка может быть передана дальше как неоднозначная):

1. Fuzzy tie (слишком близкие top1 и top2).
2. Лучший score попадает в review-зону.

## 8. Побочные эффекты и технические взаимодействия

### Чтение из внешних компонентов

1. Поиск кандидатов в runtime-кэше (точный и fuzzy blocking).
2. Чтение scoped runtime-state дедупликации источника.

### Запись во внешние компоненты

1. Запись scoped runtime-state дедупликации источника (fingerprint по каноническому ключу).

### Локальные побочные эффекты

1. Изменение внутренней таблицы "уже встреченных" fingerprint.
2. Очистка локальной таблицы перед потоковой обработкой.
3. Мутация метаданных выходного контейнера при hard-drop (добавление причины drop).

### Что стадия не делает (важно для границ)

1. Не пишет identity-индекс для целевой стороны.
2. Не создаёт pending-связи.
3. Не выполняет транзакции.
4. Не пишет логов.

## 9. Влияние конфигурации (DSL/spec/compiler options/runtime settings)

### Конфигурация правил идентичности

Влияет на:

1. Какие поля доступны для построения идентичности.
2. Порядок fallback между правилами.
3. Какое поле считается первичным в идентичности.

Результат влияния:

1. Изменяет точность exact-поиска.
2. Изменяет форму канонического ключа source-dedup.
3. Изменяет downstream-трассировку строки.

### Конфигурация игнорируемых полей fingerprint

Влияет на:

1. То, что считается "одинаковым" состоянием при source-dedup.
2. То, какие поля downstream сможет использовать для быстрого пропуска без diff.

### Конфигурация source-dedup

Влияет на:

1. Включён ли вообще контроль дублей в источнике.
2. Поведение при конфликте duplicate vs error/warn.
3. Семантику hard-drop внутри стадии.
4. Технический нюанс реализации: декларативный параметр политики для \"обычного дубля\" присутствует в конфигурации, но текущая runtime-реализация фактически всегда выдаёт warning + hard-drop для duplicate (без переключения режимом).

### Конфигурация fuzzy/scoring

Влияет на:

1. Включена ли scoring-ветка как fallback после exact.
2. По каким ключам собираются кандидаты (blocking).
3. Как считается score (компараторы и веса).
4. Пороговые решения accept/review/reject.
5. Поведение при tie.
6. Стоимость обработки (max candidates).
7. Объём explainability (top-k).

### Компиляционные опции для конфигурации стадии

Влияют не на runtime-решение, а на допустимость конфигурации:

1. Возможность требовать явное указание первичного поля для каждого правила идентичности.
2. Ранний отказ компиляции при нарушении ограничений конфигурации.

### Внешние параметры runtime

1. Флаг включения удалённых записей в поиск кандидатов меняет состав кандидатов для exact и fuzzy.
2. Наличие/отсутствие runtime-scope меняет область действия source-dedup (локально vs разделяемо).

### Переходная engine-обвязка

Поддерживается два способа сборки стадии:

1. Через контекст выполнения (новый путь).
2. Через разрозненные зависимости (legacy/transition путь).

Это влияет на способ инициализации, но не должно менять алгоритм принятия решений.

## 10. Что относится к бизнес-логике данных

### Ядро логики данных (основная ценность стадии)

1. Выбор идентичности по приоритетным правилам.
2. Семантика exact-match и обработка множественных кандидатов.
3. Семантика fuzzy fallback (blocking + scoring + thresholds + tie).
4. Правила интерпретации статусов сопоставления (найдено/не найдено/неоднозначно).
5. Формирование объяснимого решения (причина, score, список кандидатов).
6. Семантика source-dedup как политики качества входного батча.
7. Формирование fingerprint по нормализованному набору полей.

### Техническая поддержка стадии (полезно, но не ядро логики данных)

1. Сбор/восстановление служебного контекста строки.
2. Канонизация ссылки на строку для трассировки.
3. Работа с runtime-state дедупликации через порт.
4. Упаковка результата в контейнер конвейера.
5. Формирование диагностических объектов в формате общей системы диагностики.

### Спорные зоны (межслойное смешение)

1. Сборка желаемого состояния для следующей стадии внутри стадии сопоставления.
2. Мутация transport-like контейнера при hard-drop.
3. Размещение моделей следующей стадии в пакете текущей стадии.

## 11. Спорные и лишние ответственности (классификация + рекомендации)

### 11.1 Сборка желаемого состояния внутри стадии сопоставления

- Классификация: `Cross-layer leakage`
- Наблюдение: стадия сопоставления заранее собирает набор полей, который логически нужен следующей стадии для сравнения/применения изменений.
- Текущая ценность для стадии: позволяет сразу вычислить fingerprint и использовать те же данные в scoring, уменьшая дублирование преобразований.
- Почему это спорно/лишнее с точки зрения границ: стадия начинает зависеть от политики формирования состояния следующего шага, что связывает две стадии по бизнес-правилам, а не только по контракту результата сопоставления.
- Риск при оставлении как есть: сложнее менять границы стадий независимо; повышается вероятность регрессии при изменении правил формирования данных.
- Рекомендуемое место ответственности: отдельный общий компонент "подготовки сравниваемого представления" или перенос сборки желаемого состояния в следующую стадию с передачей только исходной строки/контекста.
- Минимальный шаг выделения: вынести вычисление fingerprint/scoring-представления в отдельный policy-компонент, который вызывается обеими стадиями.
- Приоритет: `High`
- Влияние на поведение: не должно измениться при корректном переносе.

### 11.2 Управление runtime-state дедупликации через порт внутри ядра стадии

- Классификация: `Stage technical support`
- Наблюдение: ядро стадии не только применяет политику дедупликации, но и само читает/пишет разделяемое runtime-состояние.
- Текущая ценность для стадии: вся логика дедупликации находится в одном месте, не требует внешнего оркестратора.
- Почему это спорно/лишнее с точки зрения границ: смешиваются правила дедупликации данных и детали хранения runtime-состояния прогона.
- Риск при оставлении как есть: усложняется тестирование/реиспользование алгоритма дедупликации отдельно от runtime-хранилища; ядро стадии становится менее чистым.
- Рекомендуемое место ответственности: отдельный runtime-state adapter/service, передающий ядру простую абстракцию "прочитать/записать seen fingerprint".
- Минимальный шаг выделения: инкапсулировать read/write seen fingerprint в выделенный объект стратегии дедупликации.
- Приоритет: `Medium`
- Влияние на поведение: не должно измениться.

### 11.3 Формирование диагностических объектов внутри ядра стадии

- Классификация: `Diagnostics presentation concern`
- Наблюдение: ядро стадии напрямую создаёт диагностические объекты конкретного формата (каталог, стадия, код, поле, сообщение).
- Текущая ценность для стадии: точная и контекстная диагностика формируется рядом с местом ошибки.
- Почему это спорно/лишнее с точки зрения границ: форматирование диагностик относится к кросс-срезовому concern; ядро могло бы возвращать типизированные причины, а отображение в диагностику выполнять отдельно.
- Риск при оставлении как есть: доменная логика сильнее зависит от диагностической подсистемы; сложнее менять формат/каталог ошибок.
- Рекомендуемое место ответственности: mapper причин стадии -> диагностические записи на границе use-case/engine.
- Минимальный шаг выделения: заменить прямую сборку диагностик на возврат типизированных domain-reasons для части веток.
- Приоритет: `Medium`
- Влияние на поведение: внешнее поведение не изменится при сохранении тех же кодов/сообщений.

### 11.4 Манипуляция transport-like контейнером при hard-drop

- Классификация: `Cross-layer leakage`
- Наблюдение: стадия при source-dedup не только принимает решение, но и модифицирует контейнер транспортного результата (очищает строку, пишет meta-flag, добавляет diagnostics).
- Текущая ценность для стадии: удобно и быстро интегрируется в pipeline без дополнительного оркестратора.
- Почему это спорно/лишнее с точки зрения границ: ядро стадии начинает знать о способе доставки результата по конвейеру, а не только о доменном решении.
- Риск при оставлении как есть: рост связности с форматом конвейера, усложнение переиспользования ядра вне текущего pipeline-контейнера.
- Рекомендуемое место ответственности: адаптер pipeline-обвязки вокруг ядра стадии.
- Минимальный шаг выделения: возвращать доменный признак drop + причину, а преобразование контейнера делать во внешнем слое.
- Приоритет: `High`
- Влияние на поведение: не должно измениться.

### 11.5 Совместное размещение моделей текущей и следующей стадии в одном модуле

- Классификация: `Cross-layer leakage`
- Наблюдение: в пакете стадии сопоставления размещены модели результата следующей стадии и тип операции следующего шага.
- Текущая ценность для стадии: удобно импортировать общие модели в переходный период, меньше модулей.
- Почему это спорно/лишнее с точки зрения границ: нарушается модульная принадлежность и усложняется понимание ответственности пакета.
- Риск при оставлении как есть: накапливается "точка притяжения" несвязанных моделей, растёт связность пакетов.
- Рекомендуемое место ответственности: общий модуль контрактов pipeline или пакет стадии разрешения/общих planning-моделей.
- Минимальный шаг выделения: перенести модели следующей стадии в нейтральный пакет контрактов и оставить совместимость через re-export.
- Приоритет: `Medium`
- Влияние на поведение: не должно измениться при сохранении импортной совместимости.

### 11.6 Размещение форматтера identity-ключа в пакете стадии сопоставления при использовании другой стадией

- Классификация: `Cross-layer leakage`
- Наблюдение: утилита канонического форматирования identity-ключа живёт в пакете сопоставления, но используется стадией разрешения ссылок.
- Текущая ценность для стадии: переиспользование без дублирования.
- Почему это спорно/лишнее с точки зрения границ: это общая утилита контракта identity-индекса, а не функция конкретной стадии сопоставления.
- Риск при оставлении как есть: вводит ложную зависимость другой стадии от пакета сопоставления.
- Рекомендуемое место ответственности: общий модуль identity/index key policy в доменном слое planning/cache-contracts.
- Минимальный шаг выделения: переместить утилиту в общий модуль и оставить совместимый импорт-переадресатор.
- Приоритет: `Low`
- Влияние на поведение: не должно измениться.

### 11.7 Переходная dual-init обвязка engine (контекст vs разрозненные зависимости)

- Классификация: `Transitional compatibility`
- Наблюдение: тонкая обвязка стадии поддерживает два способа инициализации одновременно.
- Текущая ценность для стадии: позволяет мигрировать постепенно, не ломая старые вызовы.
- Почему это спорно/лишнее с точки зрения границ: это не бизнес-возможность стадии, а переходная совместимость сборки runtime-объекта.
- Риск при оставлении как есть: усложнение конструкторов и тестовых сценариев, размывание канонического пути инициализации.
- Рекомендуемое место ответственности: временный compatibility-adapter над каноническим конструктором либо централизованный фабричный слой.
- Минимальный шаг выделения: оставить один канонический путь и вынести legacy-путь в отдельную фабрику/adapter.
- Приоритет: `Low`
- Влияние на поведение: не должно измениться.

## 12. Рекомендации по декомпозиции/переносу ответственности

### Приоритетные шаги (без изменения поведения)

1. **Развязать стадию сопоставления и сборку желаемого состояния**
   - Вынести подготовку сравниваемого/применяемого представления в общий policy-компонент.
   - Оставить в стадии только вызов абстракции, а не зависимость от правил следующей стадии.

2. **Вынести pipeline-specific hard-drop преобразование из ядра**
   - Ядро должно возвращать доменное решение "drop + причина + severity".
   - Обвязка конвейера должна превращать это в очистку строки и запись meta/diagnostics.

3. **Изолировать runtime-state source-dedup в отдельный сервис**
   - Ядро применяет политику дедупликации.
   - Сервис отвечает за хранение seen fingerprint (локально/scoped).

### Среднесрочные шаги

1. Вынести типы следующей стадии из пакета текущей стадии в общий пакет контрактов planning.
2. Вынести фабрикацию диагностик в mapper причин стадии.
3. Переместить политику форматирования identity-ключа в общий доменный модуль.

### Переходные меры совместимости

1. Оставить re-export старых импортов на время миграции.
2. Добавить тесты совместимости импорта для вынесенных моделей/утилит.
3. Сохранить неизменными статус/reason-коды и структуру explainability на период рефакторинга.

## 14. Проверочные сценарии (чеклист покрытия)

Ниже перечислены сценарии, которые должны быть отражены в анализе стадии. Для каждого сценария указано, чем он подтверждён: тестами, кодом или обоими.

1. Пустая/отсутствующая строка после upstream — подтверждено кодом (ветка раннего выхода), анализ отражён.
2. Нет идентичности ни по одному правилу — подтверждено кодом, анализ отражён.
3. Exact match найден (один кандидат) — подтверждено тестами и кодом.
4. Exact match конфликтует (несколько кандидатов) — подтверждено интеграционным тестом и кодом.
5. Exact match не найден, fuzzy выключен — подтверждено unit-тестом и кодом.
6. Fuzzy включен, blocking candidates отсутствуют — подтверждено кодом (отдельная ветка), анализ отражён.
7. Fuzzy accept — подтверждено unit-тестом.
8. Fuzzy review (ambiguous) — подтверждено unit-тестом.
9. Fuzzy reject — подтверждено unit-тестом.
10. Fuzzy tie — подтверждено unit-тестом.
11. Ограничение `max_candidates` — подтверждено unit-тестом.
12. Поведение `top_k` — подтверждено unit-тестом.
13. Unknown comparator fallback — подтверждено unit-тестом.
14. Source duplicate (same fingerprint) — подтверждено unit-тестом.
15. Source conflict (different fingerprint), режим warn — подтверждено кодом и конфигурационной веткой (drop + warning).
16. Source conflict (different fingerprint), режим error — подтверждено unit-тестом.
17. Scoped runtime-state дедупликации между экземплярами — подтверждено unit-тестом.
18. Каноничность dedup key при смене primary identity — подтверждено unit-тестом.

### Дополнительные сценарии, полезные для будущего контроля

1. Поведение при неизвестном blocking-ключе (сейчас silently skip) — подтверждено кодом, стоит закрепить отдельным тестом как переходный контракт.
2. Поведение при отсутствии runtime-scope и повторном использовании экземпляра вне потокового режима — подтверждено кодом, полезно зафиксировать тестом для избежания сюрпризов.
